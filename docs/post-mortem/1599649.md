# Post-Mortem: Code-Review Skill for GitHub PRs and Local Branches (AB#1599649)

**Branch:** `jbaker/reviewSkill` → `dev`
**Scope:** Add a single Claude Code skill — `.claude/skills/code-review/SKILL.md` — that performs an adversarial, multi-persona code review of either a GitHub PR (posting inline + summary comments and syncing this post-mortem's executive summary into the PR description) or the current local branch (output in chat). One net-new file under `.claude/`; no component source, tests, or public API touched, so this carries no semver/npm impact. This post-mortem ships alongside as the required non-functional second file.

## Executive Summary

This branch adds a `/code-review` Claude Code skill that encodes this repo's release conventions (ADO `AB#` traceability, Conventional Commits → semver, the post-mortem requirement, TRD deviations) into a repeatable, adversarial review. It runs in one of two modes keyed off its argument: a **PR number** reviews that pull request, posts reconciled inline + summary comments to GitHub, and mirrors this post-mortem's executive summary into the PR description, while **`local`** (or no argument) reviews the current branch against a chosen base and outputs to chat only — the destructive, outward-facing path is reachable exclusively via an explicit PR number.

The skill was hardened primarily by **running it against its own PR**: each pass surfaced genuine defects (a shell-injection-prone comment body, a case-sensitive keyword match, line-number-based comment reconciliation that duplicated after rebases, over-granted tool permissions) that were fixed in the same file. Two findings were structural to a review tool reviewing itself and required explicit carve-outs: `.claude/**` instructional prose is reviewed as ordinary content rather than flagged as prompt injection, and tooling changes are always `chore`, never a bump-triggering `feat`. The skill's unconditional post-mortem gate also fired a release-blocking finding on its own PR — **this document is that resolution**, and the rule was clarified to be waivable for pure `.claude/` tooling.

One property is worth stating plainly, because it changes how the skill should be used: the review is **adversarial and non-deterministic**, so re-running it on an *unchanged* diff keeps surfacing **new low-value nits**. The personas sample different angles each pass, the prompt is tuned to hunt rather than to certify, and "consider also…" suggestions are effectively unbounded — so a run almost never comes back truly empty. What *does* converge is the part that matters: genuine 🔴 correctness/security/regression findings drop to zero and stay there; only 🟡 nits churn. The implementation choice was **not** to chase determinism (which would blunt the adversarial value) but to add an explicit **convergence instruction**: when only marginal polish remains, the reviewer must report the diff as clean and mark remaining nits optional rather than manufacture a finding to look productive. The real stopping signal is therefore "no 🔴/🟡 you'd act on, two runs running" — not a literally empty list.

**Bottom line:** one net-new tooling file (no shippable code, no semver impact), a review procedure proven by executing it end-to-end against a live PR, and the key lesson that gates firing on *every* change need a typed exemption for the change classes they don't apply to.

## Why a Skill, and the Shape It Took

The goal was a repeatable, opinionated review that encodes this repo's release conventions (ADO `AB#` traceability, Conventional Commits → semver, the post-mortem requirement, TRD deviations) rather than a generic "look for bugs" pass. A few shape decisions drove the rest:

- **`disable-model-invocation: true` + `context: fork`.** The review is expensive and has side effects (it posts to GitHub), so it should run only when explicitly invoked as `/code-review`, in an isolated forked context — never auto-triggered by the model mid-conversation.
- **Mode inferred from one positional argument.** A number is a PR; empty or `local` is a local review. Everything else is a hard stop with a usage hint, so a typo'd PR number (`123x`) or a stray branch name can't silently run the wrong mode. `$ARGUMENTS` is normalized first (trim whitespace, strip a leading `#`, case-insensitive `local`) so forgivable inputs like `#1572` or `Local` don't fall through to the error path.
- **Destructive actions gated behind the explicit-PR path.** Local mode is guaranteed side-effect-free (no `gh` PR API, chat output only). Only a numeric PR argument can post public comments, and only after the SHA head-check confirms the checked-out code *is* the PR head.

## Non-Determinism and Convergence

A question surfaced during dogfooding: *if you keep running the skill after it reports no issues, will it keep finding new ones?* Largely **yes** — and understanding why matters for how the tool should be used.

- **The review is non-deterministic.** Each run samples the diff from different angles across the nine personas, so two clean runs won't produce identical output.
- **The prompt is tuned to hunt, not to certify.** "Be adversarial," "find any gaps," "assume every code path will be hit in production" — a model told to hunt will almost always return *something*, especially once the real bugs are gone.
- **Nits are effectively unbounded.** Clarity, comment density, test-coverage gaps, wording, "consider also handling X" — these are judgment calls with a near-infinite supply.
- **Self-review compounds it.** Reviewing a prompt/instruction file, "you could phrase this more precisely" is always available.

What actually converges: **🔴 correctness / security / regression findings drop to zero and stay there.** The churn is almost entirely 🟡 nits and doc polish. So the practical stopping signal is "no 🔴/🟡 you'd act on across two consecutive runs" — not "the list is literally empty forever."

**Implementation choice.** Rather than trying to make the review deterministic — which would blunt exactly the adversarial depth that catches real bugs — the skill now carries an explicit **convergence instruction** (in the "Do not flag" block of the review instructions). It tells the reviewer that an empty-handed pass is a correct, expected outcome; that it must not manufacture marginal nits to appear productive; and that when only low-value polish remains it should report the diff clean and mark remaining suggestions optional. This preserves depth on genuine defects while making "found nothing new" a first-class result instead of a prompt to keep digging.

## What Went Right

- **Dogfooding caught real defects.** The skill was run against its own PR repeatedly during development, and each pass surfaced genuine issues that were then fixed in the same file: a double-quoted `gh pr comment --body` that contradicted the skill's own "always use a quoted heredoc" rule; a case-sensitive `local` match that rejected `Local`/`LOCAL`; inline reconciliation keyed on exact line number (which would mark a rebase-shifted-but-valid comment stale and repost a duplicate); and unused `git show` / `git status` grants in `allowed-tools`. The tool improving itself by running on itself is the strongest signal it works.
- **Least-privilege tool grants.** Enumerating exact command shapes (`Bash(gh api repos/*/pulls/*/comments *)`, `Bash(git merge-base *)`, …) instead of `Bash(git *)` means the skill's authority is auditable from the frontmatter and can't be widened by a subverted instruction in reviewed content.
- **Idempotency designed in, not bolted on.** Hidden `<!-- claude-code-review:… -->` markers let a re-run find and reconcile its own prior comments; `--paginate` avoids missing comments past the first 30; the HTTP-422 fallback re-homes an un-anchorable finding into the summary rather than dropping it silently.
- **The self-review paradox was named and resolved.** A skill that reviews `.claude/**` will read agent-directed imperatives that look exactly like prompt injection. The guard exempts `.claude/**` instructional content from injection findings, so the skill doesn't flag its own (or a sibling skill's) prose.

## What Went Wrong / What the Iterations Missed

- **The post-mortem rule flagged its own PR.** The skill mandates a post-mortem for any change with commits, with no exemption by commit type — so on its *own* `chore` PR it emitted a release-blocking 🔴 for a missing post-mortem. This document is the resolution; the rule was also tightened so the requirement is understood to be waivable for pure `.claude/` tooling that ships nothing to npm. The general lesson: a "required for every change" gate needs an explicit escape hatch for the change classes it doesn't sensibly apply to (tooling, docs, CI).
- **Several correctness bugs shipped in the first drafts** and were only caught by re-running the skill — the double-quoted body, the case-sensitive keyword, exact-line reconciliation, over-granted tools. None would have been caught by reading the file once; they surfaced because the procedure was actually executed end-to-end against a live PR. Instruction-as-code needs the same "run it, don't just read it" discipline as real code.
- **`feat` vs `chore` for tooling was initially ambiguous.** Nothing stopped a new skill from being labeled `feat` and triggering a spurious MINOR release. An explicit rule now states that changes confined to `.claude/` are always `chore`, never a bump-triggering prefix.
- **`gh pr edit` is unusable on this org.** The executive-summary sync first used `gh pr edit --body-file`, which hard-errors with a "Projects (classic) is being deprecated" GraphQL error on any repo where classic projects are enabled (this one is) — even when only the body is edited, and with no project flags passed (upstream [cli/cli#11983](https://github.com/cli/cli/issues/11983)). It failed on every PR-mode run until it was swapped for the REST `PATCH .../pulls/<n>` endpoint, which has no Projects dependency and touches only the `body` field.
- **Write commands assumed a friendly shell.** Comment/description writes relied on `$(...)` substitution and `<<'EOF'` heredocs; the sandbox that runs the forked skill rejects both. Worse, the first heredoc fallback said "write to a temp file" but the frontmatter granted no `Write` and no shell-redirect Bash shape, so the fallback itself was unreachable. Fixed with a narrow `Write(/tmp/*)` grant and explicit two-step / temp-file instructions.
- **A pinned commit SHA in this very post-mortem drifted three times.** Amend-during-review plus squash-merge rewrote the add-commit SHA repeatedly; each review flagged the now-dangling reference. The fix was to stop pinning (identify by subject + branch + PR) *and* to teach the skill to flag self-referential SHAs so future post-mortems don't repeat it.

## Lessons

1. **Gates that fire on every change need a typed exemption.** The unconditional post-mortem requirement produced a false 🔴 on the very PR that introduced it. Scope release-gating rules to the change classes they're meant for (`feat`/`fix`/`perf` on shipped code), and say so in the rule — don't rely on the reviewer to "use judgment" every time.
2. **Tooling is not a feature.** Files under `.claude/` (skills, commands, hooks, agents, settings) have no public API and no semver impact; labeling them `feat` triggers an unearned MINOR release. Prefix by *what ships*, not by "this is new."
3. **Reconcile on identity, not on coordinates.** Matching prior review comments by exact line number breaks the moment a branch is rebased or code shifts above the anchor. Match on the finding's identity (file + issue substance) and treat the line as a soft hint, leaning on GitHub's `position: null` outdated signal for true staleness.
4. **Compare SHAs, not branch names, for "am I looking at the right code."** A branch-name check returns the literal `HEAD` in detached state and needs an `@{u}` upstream to catch "behind remote." A SHA compare handles both for free — and it's the safety check that makes it correct to post public comments.
5. **A self-referential tool must special-case itself.** A review skill that reads `.claude/**` will see agent-directed language indistinguishable from injection, and will see its own commit as a candidate `feat`. Both needed explicit carve-outs. Any tool that can be pointed at its own definition should be tested against exactly that.
6. **Execute instruction-as-code; don't just proofread it.** Every substantive defect here was found by running the skill against a live PR, not by reading the markdown. Treat a procedural skill like software: the acceptance test is a real run.
7. **An adversarial reviewer never runs truly dry — so give it permission to stop.** A non-deterministic, hunt-tuned review keeps producing nits indefinitely; only real 🔴 bugs converge. Left unchecked, that reads as "the code is never good enough." Bake in an explicit convergence rule: report a clean diff, mark remaining nits optional, and treat "no blocking issues" as a valid, expected result rather than a reason to keep digging.
8. **Prefer the REST API over `gh` porcelain for writes.** `gh pr edit` (and `gh issue view`) drag in GraphQL queries touching deprecated Projects-classic fields and hard-error on orgs that still have them enabled — regardless of the flags you pass. When a porcelain command has a narrow REST equivalent (`gh api --method PATCH repos/{o}/{r}/pulls/<n> -F body=@-`), prefer it: fewer hidden dependencies, a tighter permission grant, and it only mutates the field you send.
9. **Instruction-as-code must survive a hostile sandbox — including its own fallbacks.** The skill relied on `$(...)` and heredocs the sandbox rejected, and its first fallback assumed a capability (`Write`, or a shell redirect) the frontmatter never granted. Every shell construct a procedure depends on needs a fallback that is *itself* reachable under the declared tool grants — audit the fallback path, not just the happy path.
10. **Don't pin a mutable identifier in a document that ships with the thing it identifies.** A commit SHA in a post-mortem that rides the same branch is rewritten by every amend and the squash-merge, so it dangles almost immediately. Reference by stable handles (subject + branch + PR); chasing the SHA forward each run is wasted motion.
