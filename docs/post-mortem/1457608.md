# Post-Mortem: Loading State Visual Bugs in Combobox and Select (AB#1457608)

**Branch:** `sunMota/docs/loading`
**Scope:** Three fixes across `auro-combobox.js` and `auro-menu.js` — combobox bib does not open during async loading; select loading placeholder is vertically off-center and wider than the trigger.

## Symptoms → Lesson

| If you see... | This doc says... |
|---|---|
| User types into a combobox with async option loading, menu gets `loading` attribute, but the bib never opens to show the loading indicator | The `loading` attribute was set *after* the combobox's `updated()` cycle already ran with `menu.loading === false`. `showBib()` was never called. Fix: call `showBib()` from `handleMenuLoadingChange` when `loading` becomes `true`. |
| Loading slots (`loadingIcon`, `loadingText`) in the menu never render even though the bib opens after data arrives | Same root cause — the bib was hidden through the entire loading window, so the loading slots had no opportunity to render. |
| Select loading placeholder (spinner + text) is visually off-center vertically within the bib row | The inner wrapper `<div>` used inline formatting — `slot[name="loadingIcon"]` had `display: inline-block; margin-bottom: 0.125rem`. The margin extended the line box height beyond the icon's visual bounds, making flex's `align-items: center` appear off by the margin amount. Fix: make `.loadingWrapper` a flex container with `align-items: center`. |
| Select bib is wider than the trigger when the loading placeholder is visible | The loading placeholder `auro-menuoption` in the menu's shadow DOM never received `nocheckmark`. The `noCheckmark` propagation loop uses `querySelectorAll` on light DOM only, so the shadow DOM option kept its default `padding-left: 2.75rem` (44px) instead of the nocheckmark value of `0.75rem` (12px), making it ~32px wider than any regular option. Fix: hardcode `nocheckmark` on the loading placeholder in the menu template. |

## The Bug

### Async loading pattern breaks bib open

The expected loading flow in the `loading.html` example:

1. User types → combobox fires `inputValue` event
2. Consumer listener calls `emptyMenu()` then `menu.setAttribute('loading', 'loading')`
3. 2 seconds later, `loading` attribute is removed and menu is filled with new options
4. Bib should be visible throughout step 2–3, showing the loading spinner and text

**What actually happened:**

The bib never opened at step 2 and the loading slots were invisible. The bib only appeared after step 3 when new `availableOptions` triggered `showBib()`.

### Timing of the lifecycle

The root cause is a sequencing mismatch between the combobox's reactive update cycle and the consumer's `inputValue` event listener:

```
User types
  → handleTriggerInputValueChange()
      → handleMenuOptions()         ← availableOptions = [], menu.loading = false (not set yet)
  → updated(availableOptions)       ← menu.loading still false → showBib() NOT called
  → handleInputValueChange()
      → dispatch 'inputValue' event ← consumer listener runs HERE
          → emptyMenu()
          → menu.setAttribute('loading', 'loading')
              → auroMenu-loadingChange fires with { loading: true }
                  → handleMenuLoadingChange()  ← OLD CODE: only handled loading: false
```

By the time the consumer's `inputValue` listener runs and sets `loading`, the `updated(availableOptions)` cycle has already completed. The condition in `updated()` that calls `showBib()` is:

```js
if (this.availableOptions.length > 0 || this.menu.loading || this.noMatchOption) {
  if (this._userTyped && !this.dropdownOpen) {
    this.showBib();
  }
}
```

At the time this ran, `this.menu.loading` was `false` and `availableOptions.length` was `0` — the condition was `false`, `showBib()` was skipped.

When `loading` subsequently became `true` and `auroMenu-loadingChange` fired, the old `handleMenuLoadingChange` only handled the `loading: false` case (to re-show the bib after loading finished). There was no code path that opened the bib when loading *started*.

## Root Causes

### Combobox bib not opening

`handleMenuLoadingChange` was written assuming the only scenario it needed to handle was re-showing a bib that had been hidden *while* loading (the no-placeholder case). The with-placeholder case — where the bib should open *because* loading started — was not handled at all. The method had no branch for `event.detail.loading === true`.

The disconnect was that `showBib()` already knew how to handle both cases correctly, but it was only ever called from `updated(availableOptions)` (and on trigger click). When the `loading` attribute change came in asynchronously after `updated()` had already run, there was no mechanism to call `showBib()` again.

### Select loading placeholder width

`noCheckmark` propagation in `auro-menu.js` uses `querySelectorAll('auro-menu, auro-menuoption, ...')` which only traverses the menu's light DOM. The loading placeholder `auro-menuoption` lives in the menu's **shadow DOM** (`renderLayout()` template), so it is invisible to `querySelectorAll`. It never received `nocheckmark` and kept the default `padding-left: calc(0.75rem + 1.5rem + 0.5rem)` = 44px, while all user-provided options got `nocheckmark` → 12px. The 32px difference made the placeholder row wide enough to push the bib beyond the trigger width.

### Select loading placeholder vertical centering

The `.loadingWrapper` div used inline formatting. The icon slot had `display: inline-block; margin-bottom: 0.125rem`, which extends the element's inline box height downward (below the baseline). This raised the effective line-box height, shifting the logical center above the visual center of the icon. `align-items: center` on the menuoption's flex wrapper then centered the taller-than-visual box rather than the icon itself — producing a ~2px upward offset.

## The Fixes

### `components/combobox/src/auro-combobox.js` — `handleMenuLoadingChange`

Added a `loading: true` branch that calls `showBib()` when the input has a value and the component has focus:

**Before:**
```js
handleMenuLoadingChange(event) {
  if (!event.detail.loading && this.isHiddenWhileLoading) {
    if (this.contains(document.activeElement)) {
      this.dropdown.show();
    }
    this.isHiddenWhileLoading = false;
  }
}
```

**After:**
```js
handleMenuLoadingChange(event) {
  if (event.detail.loading) {
    if (this.input.value && this.contains(document.activeElement)) {
      this.showBib();
    }
  } else if (this.isHiddenWhileLoading) {
    if (this.contains(document.activeElement)) {
      this.dropdown.show();
    }
    this.isHiddenWhileLoading = false;
  }
}
```

`showBib()` already handles both cases: with placeholder → `dropdown.show()`; without placeholder → `isHiddenWhileLoading = true` (re-shown when loading ends, existing behavior unchanged).

### `components/menu/src/auro-menu.js` — loading placeholder template

Hardcoded `nocheckmark` on the shadow DOM loading placeholder so it always matches the padding of user-provided options in no-checkmark contexts:

```js
// Before
<auro-menuoption disabled loadingplaceholder class="...">

// After
<auro-menuoption disabled loadingplaceholder nocheckmark class="...">
```

The loading placeholder is never a selectable option, so it should never show a checkmark regardless of menu state. Hardcoding is cleaner than binding to `noCheckmark` and avoids the propagation-gap entirely.

### `components/menu/src/styles/default/style-menu.scss` + compiled CSS

Replaced inline-block layout with flexbox on `.loadingWrapper`:

```scss
// Before
slot[name="loadingIcon"] {
  display: inline-block;
  vertical-align: middle;
  margin-bottom: var(--ds-size-25);
  &::slotted(*) { margin-right: var(--ds-size-150); }
}

// After
.loadingWrapper {
  display: flex;
  align-items: center;
  justify-content: center;
  gap: var(--ds-size-150);
}
```

Flex layout removes all inline-formatting artifacts. `gap` replaces `margin-right` on the slotted icon, and `align-items: center` provides true vertical centering within the flex row.

## Lessons

1. **Event-driven async state changes that arrive after `updated()` need their own open/close path.** The combobox's bib visibility logic lives primarily in `updated(availableOptions)` — but reactive property changes on *child* components (`menu.loading`) don't trigger the parent's `updated()`. Any child state that should influence bib visibility needs to be handled in its corresponding event listener, not assumed to be covered by the next `updated()` cycle.

2. **When adding a new trigger for `showBib()`, check whether `showBib()` already handles the new case.** `showBib()` already had the correct branch logic for both the placeholder and no-placeholder loading cases. The only missing piece was a callsite. Prefer re-using the existing method over duplicating its logic in event listeners.

3. **`querySelectorAll` on a custom element's host does not reach elements in its own shadow DOM.** The `noCheckmark` propagation loop correctly updates all light DOM options but is blind to shadow DOM options created in `renderLayout()`. Any shadow DOM element that needs to reflect a host attribute must have that attribute set explicitly in the Lit template binding — or, for invariant properties (like `nocheckmark` on a non-selectable placeholder), just hardcoded.

4. **`margin-bottom` on an `inline-block` inside a flex-centered container shifts visual content upward.** The margin extends the element's box below the baseline, increasing the line-box height. The flex parent then centers the inflated box, not the visual content. Prefer flex with `gap` for icon+text rows — it makes spacing explicit and independent of baseline/margin interactions.
