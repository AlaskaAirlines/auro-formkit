# AB#1428414 — Datepicker & Input Locale Support + ISO Date Contract

**Branch:** `sunMota/localeSupport-sq` (bundles the ISO date contract + locale foundation and the locale follow-up work for month names, day names, and calendar forwarding)
**Scope:** ~40 files changed across `auro-datepicker`, `auro-input`, integration test suites, and all API demo pages. Introduced a new `locale` attribute, migrated every date-string interface from `MM/DD/YYYY` to ISO `YYYY-MM-DD`, and auto-derived date format and month/day names from the active locale.

---

### Section 1 — The Problem

The datepicker and input components had no concept of locale. The date format was always `mm/dd/yyyy` regardless of the user's language or region. Developers working on non-US products had to hard-code the `format` attribute and manually supply 12 translated month names. There was also no contract for what date string shape the component would accept or emit: internally the code parsed `MM/DD/YYYY` strings, which caused UTC-offset bugs in negative-UTC timezones in integration tests — a date string like `"01/15/2025"` fed to `new Date()` in a UTC-5 environment would resolve to the prior calendar day.

---

### Section 2 — Decision: ISO YYYY-MM-DD as the External Date Contract

The first architectural decision was to standardize all external date strings — `value`, `valueEnd`, `minDate`, `maxDate`, `calendarStartDate`, `calendarEndDate`, `centralDate`, `calendarFocusDate` — on ISO `YYYY-MM-DD`. This is the format emitted by `<input type="date">` and unambiguous across time zones because it carries no implicit local-midnight assumption.

Every API demo page (15+ HTML files) and all integration test fixtures were migrated from `MM/DD/YYYY` to `YYYY-MM-DD`. The integration test helper `getCentralMonth` was rewritten to read `centralDateObject` (a `Date` instance computed from the ISO string) rather than parsing the raw `centralDate` string, eliminating the UTC-offset failures.

The internal display format remains locale-derived (or explicit via `format`). The component is now a translator: ISO in, locale-formatted display out.

---

### Section 3 — The `locale` Attribute

A new `locale` property was added to both `auro-datepicker` and `auro-input`. On `connectedCallback`, the datepicker reads the nearest `data-locale` ancestor attribute via `DomHandler.getLocale()` so that a wrapping app-shell can set the locale once and have all children inherit it automatically.

When `locale` changes, `willUpdate` runs validation:

```js
try {
  const supported = Intl.DateTimeFormat.supportedLocalesOf([this.locale], { localeMatcher: 'lookup' });
  this._validLocale = supported.length > 0 ? this.locale : 'en-US';
} catch {
  this._validLocale = 'en-US';
}
```

`supportedLocalesOf` with the `lookup` matcher returns an empty array for locales the runtime does not support (covers syntactically valid but unknown tags like `"not-a-locale"`). The catch covers syntactically invalid strings like `"123"` that throw a `RangeError` before returning. Both paths fall back to `en-US`. `_validLocale` (not the raw `locale` property) is what gets forwarded to child inputs and the calendar.

---

### Section 4 — Auto-Derived Date Format

A utility `getDateFormatFromLocale(locale)` samples `Intl.DateTimeFormat` with a known reference date and infers the separator and field order (`mm/dd/yyyy`, `dd/mm/yyyy`, `dd.mm.yyyy`, `yyyy/mm/dd`, etc.) from the formatted output.

When `locale` changes and no explicit `format` is set — or when the current `format` matches the previous locale's default — the format updates automatically. An explicit `format` attribute takes priority and is left untouched. This means `locale="de-DE"` gives `dd.mm.yyyy` for free, while `locale="de-DE" format="mm/dd/yyyy"` keeps the override.

---

### Section 5 — Auto-Derived Month Names

When `monthNames` is not explicitly set, the calendar auto-populates 12 names using a `getMonthNamesFromLocale` helper in `auro-calendar-month.js`:

```js
const fmt = new Intl.DateTimeFormat(locale, { month: 'long', timeZone: 'UTC' });
return Array.from({ length: 12 }, (_, i) => fmt.format(Date.UTC(2000, i, 1)));
```

`Date.UTC` (rather than `new Date(...)`) is used deliberately to avoid DST-related day drift that could shift the sampled month boundary in some timezones. Explicit `monthNames` always win — setting them after a locale change preserves the override even if the locale changes again later. This mirrors the `format` precedence rule.

---

### Section 6 — Day Names and Calendar Forwarding

`_validLocale` is forwarded through the component tree: datepicker → calendar → calendar-month → calendar-cell. Calendar-month derives narrow day-of-week names from `Intl.DateTimeFormat` using the same locale, anchoring on `Date.UTC(2025, 0, 5)` (a known Sunday) and iterating seven days to avoid DST-induced day drift. The week always starts on Sunday regardless of locale (Sunday-first is the Alaska Airlines product requirement), so the locale only affects the displayed narrow names, not the column order.

### Section 7 — Object Getters for All Date Properties

Every date property that previously returned a raw string now has a companion `*Object` getter:

```js
get minDateObject() {
  return this.minDate && dateFormatter.isValidDate(this.minDate)
    ? dateFormatter.stringToDateInstance(this.minDate)
    : undefined;
}
```

The same pattern applies to `maxDateObject`, `centralDateObject`, `calendarFocusDateObject`, `calendarStartDateObject`, and `calendarEndDateObject`. These return `undefined` for invalid or unset dates, giving call sites a safe object to work with instead of defensive string parsing scattered across the codebase.

---

### Section 8 — `AuroInputUtil` and the Input Component

A new `AuroInputUtil` object was extracted to hold date formatting utilities shared between the input and the datepicker. It exposes three functions:

- `toISOFormatString` — re-exported from `auro-library`'s `dateFormatter`; converts a `Date` instance to a `YYYY-MM-DD` string.
- `formatISODate(isoStr, format)` — the inverse direction: takes a `YYYY-MM-DD` string and a target display format (`mm/dd/yyyy`, `dd/mm/yyyy`, `yyyy/mm/dd`, etc.) and returns the formatted display string. Returns `undefined` for invalid inputs rather than throwing.
- `toFormattedValue` — bound through `AuroInputUtilities` so the input can reuse the existing format-token replacement logic without instantiating the class itself.

This keeps the input component free of datepicker-specific knowledge while still supporting the ISO contract.

The `auro-input` component gained a `locale` attribute that forwards `_validLocale` to its internal display logic. `format` on the input remains the display format; locale auto-derives it unless overridden.

`date-fns` was added as a dependency to back the ISO string parsing and formatting utilities.

---

### Section 9 — Removing date-fns from Calendar Components

`auro-calendar-cell` previously imported `format` and `startOfDay` from `date-fns` and expected its `locale` property to be a date-fns locale _object_ (e.g. `enUS` imported from `date-fns/locale`). This created a tight coupling: the locale had to be imported as a JS module and threaded down as an object rather than a plain BCP 47 string, making it impossible to accept a locale attribute from HTML.

The fix replaced date-fns usage with native `Intl.DateTimeFormat`:

```js
// OLD — required a date-fns locale object
format(date * 1000, 'PPPP', { locale });

// NEW — accepts any BCP 47 locale string
new Intl.DateTimeFormat(this.locale, {
  weekday: 'long', year: 'numeric', month: 'long', day: 'numeric'
}).format(new Date(date * 1000));
```

The `locale` property type changed from `Object` to `String`, and `get locale()` now returns `this._locale || 'en-US'` instead of a date-fns locale object. This unblocked the end-to-end locale string flow: datepicker `_validLocale` → calendar `localeCode` → calendar-month `localeCode` → calendar-cell `locale`. `auro-calendar-month` underwent the same migration — `localeChanged()` replaced `this.locale.localize.day(int, ...)` (date-fns object API) with `Intl.DateTimeFormat`-derived narrow weekday names.

A local `startOfDay` helper (three lines of pure JS) replaced the imported date-fns function to avoid keeping the dependency just for that one utility.

### Section 10 — `referenceDates` Comparison and the `convertWcTimeToDate` Timezone Fix

Two separate date-string bugs were fixed as part of the ISO migration.

**`referenceDates` slot name comparison (`auro-calendar-cell`):**

Slot names for date-specific content are encoded as `MM_DD_YYYY`. The old comparison converted them back to slashes and compared against `MM/DD/YYYY` strings:

```js
// OLD — compared against MM/DD/YYYY strings
const compareDateStr = dateStr.replace(/_/gu, '/');
return referenceDates.includes(compareDateStr);
```

After the ISO migration, `referenceDates` values are ISO strings. The fix parses the slot name components and reassembles as ISO:

```js
// NEW — compares against YYYY-MM-DD strings
const [month, day, year] = dateStr.split('_');
const cellISO = `${year}-${month.padStart(2, '0')}-${day.padStart(2, '0')}`;
return referenceDates.includes(cellISO);
```

**`convertWcTimeToDate()` (`auro-datepicker`):**

The old implementation used `toLocaleDateString('en-US', ...)` to convert a Unix timestamp to a date string. This returns `MM/DD/YYYY` on most platforms but the output is locale- and engine-dependent, and in some environments the result differed by one day due to timezone offset applied during the conversion. The fix replaces it with `dateFormatter.toISOFormatString(new Date(time * 1000))`, which produces an unambiguous ISO string regardless of the host timezone.

### Section 11 — Adversarial Code Review (Discussion #1526)

After the branch merged, an adversarial review of the `auro-input` component was posted at [AlaskaAirlines/auro-formkit#1526](https://github.com/AlaskaAirlines/auro-formkit/discussions/1526). The review identified the following issues:

**High severity:**
- The package version was not bumped despite the ISO date string being a breaking change to the external API. (Noted as handled by the formal release process.)
- The date-format hint announcement for assistive technology was silently broken: `setCustomHelpTextMessage` / `dateFormatMap` was still present, but a case-mixed `format` attribute (e.g., `format="MM/DD/YYYY"`) missed the map lookup, leaving `setCustomValidityForType` unset and the AT hint dead.
- Migration guidance for the ISO format requirement was scattered, and non-ISO inputs failed silently without a console warning.

**Medium severity:**
- Locale changes could overwrite an explicitly-set `format` due to missing "user-set" flag logic.
- The two-digit year interpretation changed behavior without dedicated test coverage.
- `AuroInputUtil` export was undocumented in the API reference.
- `aria-pressed` was missing from the show-password button.
- DST and leap-year edge cases lacked test coverage.

**Low severity:**
- `_rawMaskValue` was written in two places but never read — dead code.
- Silent fallback to `en-US` when locale was undefined could confuse developers expecting an error.

### Section 12 — Fixes and Refactoring After Merge

The following changes were made in response to the code review and subsequent testing:

**`formatISODate` now returns `undefined` for `Invalid Date` and `null` (commit `afb0131`):**
`stringToDateInstance` returns an `Invalid Date` object for malformed strings and `null` for non-string input — it does not throw. Without an explicit guard, `getFullYear()` on `Invalid Date` returned `NaN`, and `String(NaN).padStart(4, '0')` produced `"0NaN/0NaN/0NaN"` instead of `undefined`. A `!(date instanceof Date) || Number.isNaN(date.getTime())` check was added before any field access.

**`format` attribute normalized to lowercase on `connectedCallback` (commit `8258096`):**
The `dateFormatMap` lookup in `setCustomHelpTextMessage` is keyed on lowercase tokens (`"mm/dd/yyyy"`). A case-mixed attribute like `format="MM/DD/YYYY"` missed the map, leaving `setCustomValidityForType` unset and the AT date-format hint broken. Normalizing `this.format` to lowercase once in `connectedCallback` fixed the lookup for all attribute-supplied values.

**`aria-pressed` added to show-password button (commit `434889f`):**
The show-password toggle button was missing `aria-pressed`, which is required for toggle-button state to be announced by screen readers. Added with test coverage asserting the attribute reflects the visible/hidden state.

**`*Object` getter backing fields removed from `base-input` (commit `ae8489c`):**
The `_valueObject`, `_minObject`, and `_maxObject` private backing fields — and the `_computeDateObjectFallback` helper — were removed. The getters now compute directly from the ISO string on every read (identical to the datepicker pattern from Section 7). This eliminated a lifecycle race: a parent could call `input.validate()` from inside its own `updated()` before the input's `updated()` had run `syncDateValues()`, leaving the backing fields `undefined` and silently flipping range checks to valid. Pure computed getters have no timing dependency.

**`maskInstance` nulled after `destroy()` (commit `090c24f`):**
After calling `maskInstance.destroy()`, the reference was left non-null. When the new `maskOptions.mask` was falsy (e.g., type switched from `credit-card` to `text`), the IMask reassignment was skipped — but downstream writes still routed through the destroyed instance. Fixed by assigning `this.maskInstance = null` immediately after `destroy()`.

**`isValidPartialDate()` added and hardened (commits `f2fdc17`, `80d999d`):**
A new `isValidPartialDate(value, format)` method on `AuroInputUtilities` validates single-component and multi-component partial date formats (`dd`, `yy`, `yyyy`, `mm/yyyy`, etc.) that are not full ISO dates. Single-component formats use regex + integer range checks; multi-component formats use a `date-fns` parse + round-trip. The `yyyy` range was clamped to 1900–2100 (not 1–9999), the `from` bounds for `yy`/`dd` were corrected, and the `date-fns` reference date was fixed to the 1st of the current month to prevent day-overflow rollover when parsing month-only formats (e.g., `02/2025` parsed with day=31 would roll into March).

**`selectionStart` read guarded against `InvalidStateError` (commit `77bb36d`):**
Reading `selectionStart` on input types that don't support text-selection (e.g., `number`, `email` in some browsers) throws `InvalidStateError` before anything useful can happen. The read was moved inside the `setSelectionInputTypes` gate and wrapped in a try/catch that returns early on any error, preventing a crash from reaching the `updateComplete` callback.

**Range classes reapplied after month navigation (commit `cbf4422`):**
When navigating months in a range-selection datepicker, `clearRangePreviewClasses()` removes `inRange`, `lastHoveredDate`, and `rangeDepartDate` imperatively from the cell's button. Because Lit's `classMap` tracks its own previous state, a subsequent re-render with the same class value emits no delta — the class stays absent from the DOM even though `classMap` thinks it is applied. A new `applyCommittedRangeClasses()` method re-toggles the committed-range classes (`inRange`, `rangeDepartDate`, `rangeReturnDate`) imperatively after each month navigation, resyncing the DOM with the actual committed range. It delegates to the same `isInRange` / `isDepartDate` / `isReturnDate` helpers used by `renderCellButton`, so the two paths cannot drift.

**Open item — `_rawMaskValue` dead code:**
`_rawMaskValue` is written in two places in `base-input.js` but never read. Identified in the adversarial review; not yet removed.

### Key Lessons

1. **ISO as the external contract eliminates a whole class of timezone bugs.** Parsing display-format strings (`MM/DD/YYYY`) with `new Date()` produces UTC-midnight dates that shift a day in negative-UTC timezones. ISO strings with no time component fed to a proper date library are unambiguous.

2. **`Intl.DateTimeFormat.supportedLocalesOf` is the right locale validator.** `new Intl.DateTimeFormat(locale)` does not throw for all invalid locales. `Intl.getCanonicalLocales` only validates BCP 47 syntax, not runtime support. `supportedLocalesOf` with `localeMatcher: 'lookup'` returns an empty array for unsupported locales and throws only for syntactically malformed strings.

3. **Separate `_validLocale` from the raw `locale` attribute.** The locale the user sets may be unsupported; child components should always receive the validated fallback, not the raw input.

4. **Auto-derive, explicit overrides always win.** Both `format` and `monthNames` follow the rule: auto-derive from locale unless the developer explicitly set the value. Encode this precedence in `willUpdate` by checking `changedProperties.has('format')` before clobbering the format.

5. **Locale must flow as a BCP 47 string, not a library object.** Passing date-fns locale objects through component properties works in JS but breaks HTML attribute binding. Migrating to `Intl` APIs and plain string locales makes the entire tree accept locale from HTML, from JS, and from ancestor `data-locale` attributes without any special wrapping.

6. **`toLocaleDateString` is not a safe date serializer.** Its output format is locale- and engine-dependent and applies the host timezone offset, which shifts dates by a day in negative-UTC environments. Use a dedicated ISO serializer (`dateFormatter.toISOFormatString`) any time you need a stable date string from a timestamp.

7. **`*Object` getter companions clean up defensive parsing.** Replacing scattered `dateFormatter.isValidDate(this.foo) ? dateFormatter.stringToDateInstance(this.foo) : undefined` inline calls with a single named getter makes intent clear and gives one place to add tests for the valid, invalid, and unset cases.

8. **Locale inheritance via `data-locale` reduces boilerplate.** Rather than requiring every component to receive `locale` as an explicit attribute, reading it from the nearest ancestor attribute lets a single app-level declaration propagate automatically — the same pattern browsers use for `lang`.
