## Symptoms → Lesson

| If you see... | This doc says... |
|---|---|
| WTR reports `0 passed, 0 failed` after the full 120s `testsFinishTimeout` | Most likely chai's diff inspector deadlocking on `expect(null).to.equal(domElement)` with a custom element. Not a Lit cascade. Rewrite affected assertions as `expect(a === b).to.be.true` so they fail fast. |
| ~41 mobile combobox tests fail with `waitUntil` timeouts on `el.shadowRoot.activeElement === el.inputInBib` | WTR's `rootDir` is **CWD-relative**, not config-relative. Running from repo root 404s design-token CSS, mobile fullscreen never activates, `inputInBib` never receives focus. Run from the component directory. |
| React framework tests fail with `Cannot set property hasLoadingPlaceholder of #<AuroMenu> which has only a getter` and combobox `firstUpdated` never finishes | React 19's `'propName' in element` check picks up read-only getters and attempts assignment. Remove the dead JSX attribute from React fixtures — Svelte silently no-ops via `setAttribute`, React throws. |
| Dynamic-menu Swap-values demo pegs the main thread for ~5s after a swap | Two writers (`setMenuValue` and the `auroMenu-selectedOption` listener) are fighting over `combobox.value`. Use a one-shot `_pendingMenuValueSync` flag set by `setMenuValue` and consumed by the listener on its first event. Promise-based clears race the cascade — only the one-shot works, with a `setTimeout(0)` backup-clear. |
| `combobox.value` oscillates between two values after a programmatic swap | Same root cause as above — the listener's `this.value = this.optionSelected.value` write echoes back through `setMenuValue`. Echo detection via `_pendingMenuValueSync`. |
| Combobox's `auroMenu-selectedOption` handler reads `undefined` from `event.detail.options` / `stringValue` / `label` | Post-revert menu dispatches `{ source }` only. Read state from `this.menu.optionSelected` and `this.menu.value` directly, not from `event.detail`. |
| `combobox.input.value` contains leading whitespace, HTML indentation, or duplicates the `[slot="displayValue"]` text (emoji / icon) | The listener wrote `this.optionSelected.textContent` directly. Use `getOptionLabel(option)` — clone the option, strip the displayValue child, normalize whitespace. |
| Bib auto-opens on a no-match-side combobox after a programmatic swap, without user interaction | `handleMenuOptions()` is being called during the programmatic sync, which triggers the unconditional `availableOptions.length === 0 && this.noMatchOption → showBib()` clause in `updated()`. Suppress with a `_programmaticFilterRefresh` one-shot consumed by the availableOptions branch. |
| Bib filter is stale after a programmatic value swap (left input shows "f" but bib lists all options) | `_syncingDisplayValue` is correctly suppressing re-entrancy but also suppressing the legitimate filter refresh. Capture `isEcho = this._pendingMenuValueSync` at the top of the listener (before the cascade-fix code consumes it) and call `handleMenuOptions()` when `isEcho` is true. |
| Credit-card combobox stuck at `validity: tooShort` after selecting a valid option | `inputInBib` is dispatching `auroFormElement-validated` with stale `validity`, the listener writes it to `this.input.validity`, then `validation.js:380` reads back the polluted state. Fix the root cause with symmetric trigger↔bib sync in `updateTriggerTextDisplay` — once both inputs always agree, the `bibOpen`-aware listener filter and the explicit `this.validity = undefined` in `reset()` become dead code. |
| Bib helpText shows the error message even though the input has no error styling | Same root cause as the validity bug — `inputInBib` is on a stale value driving stale `validity` → stale `errorMessage` → helpText. Symmetric sync resolves it. |
| Fullscreen mobile: type "Apples", Tab to select, backspace 6 times only deletes one character | `showModal()` steals focus, `trigger.inert = true` removes it, `doubleRaf` (~32ms) is too slow for the recover. Add a single `requestAnimationFrame` (~16ms) before the `doubleRaf` and a native-input value sync (`bibNativeInput.value = this.input.value`) for the Lit-property-to-native async gap. |
| Setting `combobox.value = 'invalid-option'` twice in a row in `behavior='filter'` doesn't clear the value the second time | The post-revert no-match else branch bypassed `setMenuValue` entirely. For `behavior='filter'`, route through `setMenuValue` so the failure cascade fires every time — the same-value guard would otherwise swallow the second call. |
| Framework tests report a regression that doesn't reproduce in wtr after a source change | Framework apps consume `dist`, not `src`. Rebuild before re-running. |
| Test passes vacuously even though browser logs show 50+ `performUpdate` calls | `await el.menu.updateComplete` resolves before queued cascade updates run. Add a circuit-breaker on `LitElement.prototype.performUpdate` (throw at >50 calls) plus a 50ms macrotask settle before the assertion. Without the breaker, a regression looks identical to WTR infra flake. |

## What Was Not in the Plan but Also Shipped

A cascade of behavioral bugs surfaced once the pre-5.9 event payloads were back in play and the cross-component sync paths reverted. Each required source changes and a locking regression test.

1. **`auroMenu-selectedOption` payload shape change broke 56 tests at once.** The reverted menu sends `{ source }` only; the 5.9 handler read `event.detail.options/stringValue/label`. Fix: read state directly from `this.menu`.
2. **Re-entrant input events from `base-input.notifyValueChanged()`.** Setting `this.input.value` programmatically dispatches a native `input` event into `handleInputValueChange`, which can re-write `this.input.value` and loop. Fix: `_syncingDisplayValue` flag, set only when a real write is queued, cleared after `updateComplete`. Coarse-flagging (set unconditionally) broke 13 small-viewport tests — the guard is load-bearing.
3. **State oscillation in dynamic-menu swap.** Two writers (`combobox.value` and `auroMenu-selectedOption` listener) fought over `combobox.value` after `swapValues()`, pinning the main thread at ~125 Lit `performUpdate` calls. Fix: one-shot `_pendingMenuValueSync` flag set by `setMenuValue` and consumed by the listener on its first event. Backed up by `setTimeout(0)` so the flag doesn't outlive a no-op `setMenuValue`. Promise-based clears raced the cascade — only the one-shot worked.
4. **Stale bib filter after programmatic value swap.** `_syncingDisplayValue` correctly suppressed re-entrancy but also suppressed the legitimate `handleMenuOptions()` refresh. Fix: capture `isEcho = this._pendingMenuValueSync` at the top of the listener (before the cascade-fix code consumes it) and call `handleMenuOptions()` when `isEcho` is true. Paired with `_programmaticFilterRefresh` so the no-match auto-open clause in `updated()` skips programmatic syncs.
5. **`input.value` leaked `displayValue` slot text and HTML indentation.** The listener wrote `this.optionSelected.textContent` into `input.value`, double-counting the `[slot="displayValue"]` content and including whitespace. Fix: `getOptionLabel()` helper clones the option, strips the displayValue child, normalizes whitespace.
6. **Validation cascade pollution.** The combobox listener applied `inputInBib`-dispatched `auroFormElement-validated` events to `this.input.validity`. When `inputInBib` was stale (un-focused, value undefined), it dispatched `validity = undefined`, polluting the trigger's just-computed `valueMissing`. `validation.js:380` then read back the polluted state. Fix: symmetric `updateTriggerTextDisplay` that writes both inputs, plus a bail in `handleInputValueChange`'s bib branch on `_syncingDisplayValue`. After the symmetric sync, the `bibOpen`-aware listener filter and the explicit `this.validity = undefined` in `reset()` were verified dead and removed.
7. **Fullscreen focus loss during transition.** `showModal()` steals focus; `trigger.inert = true` removes it; `doubleRaf` (~32ms) was too slow to recover. Fix: an additional single `requestAnimationFrame` (~16ms) before the `doubleRaf` plus a native-input value sync (`bibNativeInput.value = this.input.value`) to handle Lit's async property-to-native sync gap.
8. **`auro-select`'s `setMenuValue` bypass.** Post-merge of Sub-PR 4, the `setMenuValue` path was writing `this.menu.value = value` directly, skipping `menu.selectByValue()`'s stringify guard and storing arrays unstringified in `auro-form`. Fixed in this branch on the select side (1-line). Documented in [[1560488]] as item 6.
9. **`auroMenu-selectedOption` framework test asserted on `{ value }` payload.** The post-revert detail is `{ source }` only. Rewrote the test to read `menu.value` directly. Briefly extended the menu's payload and broke 6 wtr rows — the `{ source }`-only contract is deliberate.
10. **Async-option load + early `value` bind race.** Framework tests render `<auro-select value="bar">` and load `<auro-menuoption>` children 300ms later. Pre-5.9 menu hit the no-match path before items arrived and `auroMenu-selectValueFailure` nulled `select.value`. Fix in `auro-menu.js`: defer failure when `items` is empty; on slotchange, re-trigger value matching against newly-arrived items.

## What the Plan Got Right
- `comboboxKeyboardStrategy.js` was architecturally independent of MenuService as #1463 predicted.
- The "post-plan bug-fix commits will be lost" warning in #1463 §8 Q4 was accurate. Most were re-ported; the freeform-swap commit (`31b22499`) needed a behavioral rewrite, not a transcription.
- Sequencing (menu → select → combobox) held — the menu/select changes from Sub-PRs 3 and 4 were stable enough to revert against.

## What the Plan Missed
- **The "filter hang" was not a Lit cascade.** Three sessions chased a re-entrant update loop. The actual cause: `expect(null).to.equal(domElement)` deadlocks chai's diff inspector for the full 120s WTR `testsFinishTimeout` when the element is a custom element with shadow DOM. Mocha never reports the failure; WTR reports `0 passed, 0 failed`. Pattern is identical to a runaway loop. Filed to memory as `project_combobox-filter-hang-is-chai`. Mechanical fix: rewrite affected assertions with strict equality (`expect(a === b).to.be.true`) which fails fast with a normal message.
- **WTR `rootDir` is CWD-relative.** `packages/config/src/web-test-runner.config.mjs` sets `rootDir: '../../'` which resolves relative to **CWD**, not the config file. From the repo root, leading-slash `/node_modules/...` paths 404, design-token CSS never loads, the `--ds-grid-breakpoint-sm` token resolves empty, `mobileFullscreenBreakpoint` returns undefined, mobile fullscreen never activates, ~41 mobile tests fail with `waitUntil` timeouts on `el.shadowRoot.activeElement === el.inputInBib`. Filed to memory as `project_wtr-rootdir-cwd`. Always run from the component directory.
- **React 19 throws on getter-only custom-element props.** React 19's `'propName' in element` check picks up read-only getters and tries `el.hasLoadingPlaceholder = true`, throwing `TypeError` in strict mode. Svelte's compiler falls through to `setAttribute` and silently no-ops. Symptom in React framework tests: 10 hard fails on `waitForFunction(combobox?.input != null)` because the throw bubbles out of the React render and the combobox child never finishes `firstUpdated`. Fix: remove dead `hasLoadingPlaceholder` JSX attribute from React fixtures — the getter derives state from the `loadingText` slot which is already present.
- **Symmetric trigger ↔ bib sync replaced compensating filters.** Before symmetric sync, a `bibOpen`-aware listener filter and an explicit `this.validity = undefined` in `reset()` compensated for the validation pollution cascade. After symmetric sync both became verifiably dead code and were removed. The cleanup is fragile — a future "consolidate flag-setting" refactor could re-introduce the cascade. Filed to memory as `project_combobox-validity-cascade` with the rationale for the `!==` guard.
- **Framework apps consume `dist`, not `src`.** A "regression" report on Session 9 was a stale dist build — rebuilding fixed it. Always confirm freshness before chasing.

## Testing Strategy Adherence
#1476 prescribed tests-first; this PR did fix-first for nearly every bug above because none were predictable from the test baseline. Each fix landed with a locking regression test in the same or next commit. Test count grew from 425 baseline → 467 final, all green.

A few regression tests required defensive instrumentation: the cascade-prevention test was passing vacuously until a circuit-breaker (`throw at >50 performUpdate calls`) and a 50ms macrotask settle were added. Without the circuit-breaker, a regression manifested as WTR's `0 passed, 0 failed` 120s timeout — the same shape as infrastructure flake.

## Outcome
3 source files changed in `components/combobox/` plus the menu async-defer patch and the select `setMenuValue` fix. 467/467 wtr in ~8.7s. Cascade dropped from ~125 Lit updates to 7. React framework 358/358, Svelte 354/2 (2 pre-existing flakes). 10+ behavioral bugs landed beyond the planned scope, each with a regression test.

## Recommendations for the Remaining Sub-PR (`@lit/context` removal)
1. Verify no remaining imports across all components, packages, and apps before deleting the dependency. The previous sub-PRs each missed a downstream consumer in initial discovery.
2. When a wtr suite reports `0 passed, 0 failed` after a long delay, suspect chai's diff inspector before suspecting a Lit cascade. Quick check: does any failing assertion compare a DOM element to a non-DOM value?
3. Always run wtr from the component directory. Run-from-root failures look like real regressions and burn diagnosis time.
4. Frontend framework consumers diverge in how they hand off props to custom elements. Read-only getters are a React 19 footgun. When restoring or removing a property on a public element, audit React fixtures explicitly.

## Post-Rebase Regression Analysis (2026-06-09)

**Branch:** `rmenner/formkit-v6/revert-menu` (continuation of the v6 revert series, post-upstream-rebase)

The v6 revert branch was rebased onto an upstream that pulled in two large changes: the base-input rewrite (+434 lines) and the `feat: add locale support on datepicker and input` commit (`9c515c8`) which rewrote `validation.js`'s date path to use `valueObject` instead of `dateAndFormatMatch`/`validDateStr`. The credit-card path in `validation.js` was unchanged on paper, but its surrounding dispatch and re-entrancy behavior was not.

After two infra fixes that cleared most failures (`npm install` for the new `auro-library@5.13.0` dateFormatter export, and `npm run build:force` to break a stale `FULL TURBO` cache so `components/input/dist/index.js` actually picked up the new validation), 8 failures remained:

| Suite | Count | Cause |
|---|---|---|
| auro-combobox | 2 | The cleanup at sub-pr-5 close became load-bearing again — see below |
| auro-dropdown | 6 (3 × 2 viewports) | desktopModal focus-trap regression, on `origin/dev` too — coincidentally surfaced but unrelated to sub-pr-5 |

### 1. The fragile cleanup warning landed — for a different reason

[#62](#what-the-plan-missed) (item 4 above) warned: *"a future 'consolidate flag-setting' refactor could re-introduce the cascade."* The cascade did re-introduce itself, but no one refactored sub-pr-5. The upstream base-input/validation rewrites changed the input's validate-dispatch pattern enough that the symmetric `updateTriggerTextDisplay` sync no longer fully protected against bib-input pollution of `this.input.validity`.

**What the new code looks like.** Two compensations are back, in different shapes than the ones we removed at sub-pr-5 close:

- **Listener filter (replaces the `bibOpen`-aware `authoritativeInput` check):** the `auroFormElement-validated` listener now bails when `evt.composedPath()[0] === this.inputInBib` and the bib input isn't also the trigger input. `composedPath()[0]` is the original dispatcher pre-shadow-retargeting, so this isolates events that actually originated from the bib input even though `target` shows up as the combobox. Source: `components/combobox/src/auro-combobox.js` listener body in `configureCombobox()`.

- **Explicit `reset()` clear:** the trailing `this.validity = undefined; this.errorMessage = ''; this.input.validity = undefined; this.input.errorMessage = ''` is back at the end of `reset()`. The credit-card listener filter blocks the bib input from wiping the trigger's residual `valueMissing` — which means `validation.reset(this)`'s internal `validate()` now reads back the trigger's leftover `valueMissing` via `validation.js:380` and the reset doesn't actually clear. Explicit clear after the validate is the only way to leave the component in a "no validity" state. Source: `components/combobox/src/auro-combobox.js` `reset()`.

The two changes are coupled — landing either one alone regresses the other test. The credit-card `tooShort` regression and the `reset method clears the value and validity state` test are direct duals.

**Test count:** combobox went 467 (sub-pr-5 close, pre-rebase) → 465 (post-rebase, 2 fails) → 467 (post-fix). One existing test (`setInputFocus calls setClearBtnFocus and validate when persistInput and menuoption focused`, added at sub-pr-5 close) needed a `querySelector(':focus')` stub because the combobox's `focusin → this.focus()` redirect synchronously moves focus away from the menuoption the test had just focused, so the production branch's `el.querySelector(':focus')` reads null. The branch itself works in production for the keyboard-nav-in-fullscreen case; the test had been relying on focus semantics the input rewrite quietly changed.

### 2. Dropdown desktopModal focus-trap — coincidentally surfaced

Not in sub-pr-5 scope, but uncovered by the same green-the-suite work and worth recording here so it isn't re-discovered later. `git diff origin/dev..HEAD -- components/dropdown` is empty; the regression exists on `dev` too. The desktopModal focus-trap test suite (test/auro-dropdown.test.js:899, :975, :1124, × 2 viewports = 6 failures) was failing.

**Root cause.** `auro-dropdown.js`'s `updateFocusTrap()` installs `_bibTabHandler` and schedules a RAF to move initial focus into the bib. The RAF's `focusables[0].focus()` triggers `AuroFloatingUI.handleFocusLoss` (registered on document at `floatingUI.mjs:607`), whose guard `element.matches(":focus-within")` *returns false* even though `element.contains(document.activeElement)` is true. The mismatch is a popover-top-layer style-recalc timing artifact — `:focus-within` hasn't propagated yet at the moment the focusin handler runs. `handleFocusLoss` proceeds to `hideBib`, closes the popover, and tears down `_bibTabHandler` before the test's synthetic Tab keydown arrives. Capture-phase probe confirmed Tab reaches the dropdown with `hasHandler: false`.

The desktopModal tests that "passed" (test 860, 1091) only did so vacuously — test 860's assertion `el.querySelectorAll('button')[0]?.getRootNode() === active?.getRootNode()` is `document === document` for any two attached nodes, and test 1091 explicitly focuses a `<button slot="trigger">` which is light-DOM-contained by `el` regardless of what the handler does. The entire feature was broken; the test suite missed it.

**Fix.** While the desktopModal trap is installed, the dropdown owns its own focus management — AuroFloatingUI's auto-hide-on-focus-loss must be suppressed. `updateFocusTrap()` now sets `this.noHideOnThisFocusLoss = true` at install (saving the prior value to `this._priorNoHideOnFocusLoss`) and restores at cleanup. Click-outside and ESC dismissal paths are unaffected (those run through `clickHandler` and `auro-bib-cancel` respectively, not through `handleFocusLoss`).

**Test count:** dropdown 338/344 → 344/344. No new tests required — the existing 6 failing tests serve as the regression lock.

### Outcome (Post-Rebase)

Full forced rebuild: `npx turbo run test --force` → 28/28 tasks successful, 0 cached, 0 failures across all component suites.

- `components/combobox/src/auro-combobox.js` — listener filter (+10 lines), reset() explicit clear (+9 lines).
- `components/combobox/test/auro-combobox.test.js` — querySelector stub on the menuoption focus test (~9 lines net).
- `components/dropdown/src/auro-dropdown.js` — focus-loss suppression install + cleanup (+17 lines).

### Lessons

- **The original "verified dead code" cleanup at sub-pr-5 close was correct against the pre-rebase upstream and wrong against the post-rebase upstream.** "Dead code" verified against one snapshot of dependencies is a snapshot, not a property. The fragility warning was right; the prediction of *what* would trip it (a local refactor) was incomplete — an upstream rewrite of base-input was equally capable.
- **Turbo's `FULL TURBO` cache will gladly serve stale dist artifacts after a dependency upgrade.** `npm run build:force` is the safety. The handoff document called this out explicitly; following it saved the diagnosis.
- **Vacuous test assertions hide latent regressions across rebases.** The dropdown desktopModal feature was broken on `origin/dev` and went undetected because two of the six tests in that suite asserted on conditions that trivially hold. When adding regression tests for focus-management features, the assertion must actually distinguish the "trap held" case from the "trap was torn down and focus escaped to body" case — `el.contains(document.activeElement)` works; `getRootNode() === getRootNode()` does not.

## Imask cursor throw + Diners regex (2026-06-09 evening)

WTR was 28/28 green after the post-rebase recovery above. The combobox `customize` demo (credit-card combobox) was throwing an uncaught `TypeError` on real-mouse menuoption clicks that the WTR suite did not catch. Diagnosed and fixed in the same session.

### Symptom

Real-mouse repro on `customize.html#inputType`:
1. Click the credit-card combobox to focus
2. Type "3"
3. Real mouse-down/up on the first visible option ("3400 000000 00000")

Console produced two simultaneous throws:
```
TypeError: Cannot read properties of undefined (reading 'selectionEnd')
    at get cursorPos (imask/.../input.js:147)
    at InputMask.alignCursor
    at InputMask.updateControl
    at set value
    at AuroInput.configureAutoFormatting (base-input.js:984)
    at AuroInput.updated (base-input.js:778)
```

WTR didn't catch it because (a) the throw is in a Promise-style synchronous chain inside imask that mocha doesn't surface as an unhandled rejection by default, (b) scripted `option.click()` does not reproduce — only real `page.mouse.down/up` events do.

### Root cause

Re-entrant `configureAutoFormatting` via the patched-setter's synthetic input event:

1. `configureAutoFormatting` creates `mask_v1`, then runs `mask_v1.value = existingValue` (base-input.js:984).
2. Inside imask, `set value` → `updateControl('auto')` writes `this.el.value = newDisplayValue` (input.js:185) **before** calling `alignCursor()`.
3. The `el.value = …` write goes through `patchInputEvent`'s setter (base-input.js:687-715). With `skipNextProgrammaticInputEvent` already consumed by the earlier `inputElement.value = ''` (line 964), the setter dispatches a synthetic `input` event.
4. The synthetic event fires `handleInput` → `processCreditCard()` → which mutates `this.format` and calls `configureAutoFormatting` re-entrantly.
5. The re-entrant call calls `this.maskInstance.destroy()` — destroying `mask_v1` (sets its `el` to `undefined` per imask:345) and constructs `mask_v2`.
6. Stack unwinds back to the outer `updateControl`'s `alignCursor()`. `cursorPos` getter reads `mask_v1.el.selectionEnd` → `mask_v1.el` is `undefined` → **throw**.

The two simultaneous throws come from trigger input and bib input each hitting the same re-entrancy (both have `type="credit-card"` and their own `processCreditCard` running from the render template).

### Why the prior reproductions appeared inconsistent

- The 14-digit Diners option `3000 000000 0000` reliably hit it. The 16-digit Visa option `4000 0000 0000 0000` did not always — for some values the masked `newDisplayValue` happened to equal the existing `el.value`, so the `if (this.el.value !== newDisplayValue)` guard inside `updateControl` skipped the write, no synthetic event fired, no re-entry occurred.
- Focus state also mattered: the patched setter early-returns when the input matches `:focus`. The trigger input typically holds focus after dropdown close; the bib input doesn't. That's why the bib-side throw was consistent and the trigger-side throw was sporadic.

### Fix

Four coordinated changes in `components/input/src/base-input.js`:

1. **Re-entrancy guard on `configureAutoFormatting`** — `if (this._configuringMask) return;` at the top; wrap body in `try { … } finally { this._configuringMask = false }`. Prevents the destroy-mid-flight cascade in step 5 even if the synthetic event slips through.
2. **Patched-setter early-return on `_configuringMask`** — `if (component._configuringMask) return;` in `patchInputEvent`'s setter, after the existing skip-flag check. Suppresses the synthetic input event from imask's internal `el.value` write while the mask is being built.
3. **Accept/complete listener guards** — `if (this._configuringMask) return;` at the top of both handlers. Without this, the mask's own value-restoration (line 984) fires `accept` and clobbers `this.value` with whatever raw text was in `inputElement.value` at the start of the call — overwriting the value the combobox just pushed.
4. **Route credit-card value writes through the mask** — in `updated()`'s value-change branch (line 833), changed `this.inputElement.value = formattedValue` to `this.maskInstance.value = formattedValue` for `type === 'credit-card'`. Keeps imask's `_value` and `el.value` in lock-step. Without this, the mask's `displayValue` stays stale relative to the new `el.value` and the next focus/click trips imask's "Element value was changed outside of mask" warning. Scoped to credit-card so date's own `formattedValue` (raw ISO when calendar-invalid) isn't re-masked through the mm/dd/yyyy mask — that would truncate it and flip validity from `patternMismatch` to `tooShort`.

### Diners 30-prefix coverage (separate, surfaced during repro)

`matchInputValueToCreditCard`'s Diners regex was `^(?<num>36|38)`. Real-world Diners Club cards have always included the legacy `300-305` range, and the `customize.html` demo uses `3000 000000 0000` as the Diners sample. With the original regex, "30…" fell through to Default Card (4-4-4-4, formatLength 19), so the demo card displayed re-masked as "3000 0000 0000 00" and validated as `patternMismatch`. Fix: broadened regex to `^(?<num>30[0-5]|36|38)` and swapped `cardIcon: 'credit-card'` to `cardIcon: 'cc-dinersclub'` (the icon was already in `@alaskaairux/icons/dist/icons/payment/`, just unused).

### Outcome

- Real-mouse repro on customize demo: throw gone, "value changed outside of mask" warning gone, trigger displays the formatted card number, Diners icon renders, validity is `valid` for sample card numbers.
- WTR: input suite went 456 → 464 (Diners icon assertion update) → 472 (with 8 new regression tests covering: 30-prefix Diners detection, no-throw/no-warn across format changes, mask `displayValue === el.value` after programmatic write, configureAutoFormatting re-entrancy guard idempotency). Each variant runs in both default and small-viewport context.
- Full suite: 27/28 component suites green. The remaining `auro-datepicker` failure (`should show dateTo error message when dateFrom is valid`, `should clear the value and validity state when reset() is called`, × 2 viewports = 4 tests) is **pre-existing** — confirmed by stashing all changes and re-running; the failures reproduce against pristine HEAD. Not introduced by this fix.

**Files changed:**
- `components/input/src/base-input.js` — re-entrancy guard, patched-setter early-return, accept/complete guards, credit-card mask-routing, Diners regex broadening, Diners icon swap.
- `components/input/test/auro-input.test.js` — Diners icon assertion updated to `cc-dinersclub`; 4 new regression tests added (× 2 viewport contexts = 8 new test runs).

### Why WTR did not catch this in sub-pr-5 close

The credit-card cascade test at sub-pr-5 close asserted on combobox state (value, optionSelected, validity), not on console errors. The cascade *completes* (the value gets set) even when the mask throws partway through — the throw is in a side-effect path (`alignCursor` after `el.value` is already written). A `failOnBrowserError` plugin in the WTR config would be the long-term lock; for now the new "no errors or warnings across format changes" test in `auro-input.test.js` covers the regression directly.

### Lessons

- **Side-effect re-entrancy through synthetic events is invisible to unit tests until it throws.** The mask's internal `el.value = displayValue` is a no-op as far as masked semantics go, but it goes through the patched setter, which dispatches an event whose downstream handlers can mutate the mask's owner. The mechanism existed pre-sub-pr-5; it was harmless until upstream's rewrite increased the frequency of `format` mutations.
- **Don't write to a backing element while a wrapper still thinks it owns the value.** Routing the credit-card value-write through `maskInstance.value` instead of `el.value` directly is the correct shape for any masked input. The date path stays direct because the date utility computes `formattedValue` semantically (including raw-ISO fallback for calendar-invalid input) and re-feeding that through the mm/dd/yyyy mask would corrupt validity.
- **Real-world card-prefix ranges drift; regexes don't.** The Diners `300-305` range has been valid for decades. The regex `^36|38` was a snapshot of the issuing pattern someone saw on a Mastercard reference page. When the demo data accurately reflects a real card and the validator doesn't, the validator is the bug.
