# Post-Mortem: Counter Shipped Unbundled `@aurodesignsystem/auro-dropdown` in 5.12.0/5.12.1 (AB#1575423)

**Branch:** `sunMota/rollup-fix`

## What happened

`@aurodesignsystem/auro-formkit@5.12.0` (May 8) and `@aurodesignsystem/auro-formkit@5.12.1` (May 11) shipped to npm with broken `counter/dist/index.js` and `counter/dist/registered.js`. Both files contained top-level bare imports that rollup was supposed to inline:

```js
import { AuroDropdown } from '@aurodesignsystem/auro-dropdown';
import { AuroBibtemplate } from '@aurodesignsystem/auro-bibtemplate';
```

A downstream consumer hit the import-resolution failure and reported it. Counter was the only affected component; the other twelve components in the same tarball were clean. 5.11.0 had been fine.

## How it got found, and how the first hypothesis was wrong

The initial fix on this branch ([`4f2273d08`](https://github.com/AlaskaAirlines/auro-formkit/commit/4f2273d08)) flipped `preferBuiltins: false → true` in the rollup config. The reasoning: maybe `nodeResolve` was looking in the wrong place for the workspace symlink. The fix was merged but not yet published.

When this work started, the goal was to verify that fix. Reverting it locally and rebuilding — the leak didn't come back. That meant `preferBuiltins` was never load-bearing. Something else had quietly resolved the bug between the 5.12.1 publish and this investigation.

The real diagnostic ended up being a worktree at `v5.12.0`. Wiping `dist/`, rerunning `npm run build`, and inspecting `counter/dist/index.js` reproduced the leak every time. Running just counter's rollup alone (after dropdown's dist was already on disk) produced clean output. That gap told the story: order of operations.

## Root cause

A few weeks before the bug shipped, [`f623bd1cb`](https://github.com/AlaskaAirlines/auro-formkit/commit/f623bd1cb) (`chore: reduce vulnerabilities, hoist shared deps, and remove unused packages`) deleted the `dependencies` block from every component's `package.json`. The intent was to rely on npm workspace auto-linking. For most components, the commit also added an explicit `<name>#build` `dependsOn` block to `turbo.json` enumerating their workspace sibling deps:

```json
"@aurodesignsystem/auro-combobox#build": {
  "dependsOn": [
    "@aurodesignsystem/auro-bibtemplate#build",
    "@aurodesignsystem/auro-input#build",
    "@aurodesignsystem/auro-dropdown#build"
  ]
}
```

**Counter didn't get an entry.** It inherited the default `build` task, whose `^build` resolves to the empty set when the package declares no dependencies. Turbo concluded counter had no upstream builds to wait on and scheduled it in parallel with dropdown's and bibtemplate's builds.

When counter's rollup ran while `components/dropdown/dist/index.js` didn't exist yet, `nodeResolve` followed the workspace symlink (which exists fine), read dropdown's `package.json`, saw `main: "dist/index.js"`, attempted to load that file, found nothing, and returned `null` from its resolver. Rollup's default for unresolved imports: emit a warning to stderr, treat the specifier as external, leave the bare `import` in the output, exit 0. The bad dist got cached, passed every test in CI, and shipped.

The race was unstable. When dropdown happened to finish first, counter's dist came out clean. That's why the bug felt intermittent.

[`310f18afb`](https://github.com/AlaskaAirlines/auro-formkit/commit/310f18afb) (`ci: declare auro-counter build deps to prevent node-resolve race`, June 4) added the missing turbo entry. That's the actual fix. It landed three weeks after 5.12.1.

## Why CI tests didn't catch it

Workspace symlinks. Per-component WTR tests load source from `src/`, not `dist/`. Framework smoke tests load `dist/registered.js` through vite, which resolves bare specifiers against the workspace's `node_modules` — and `node_modules/@aurodesignsystem/auro-dropdown` is a symlink to `components/dropdown` regardless of whether its dist is malformed. The leaked specifier resolved fine inside the workspace.

The failure was only visible to a consumer that didn't have the workspace topology. FSC and LEM, installing from npm into their own dependency trees, did.

## What this branch adds

The root cause is already fixed on `dev`. This branch adds three defenses so the same failure class — silent rollup externalization, regardless of trigger — can't ship to npm again.

1. **`test/check-bundled-imports.mjs`** scans every `components/*/dist/{index,registered}.js` against an allowlist derived from `EXTERNAL_PACKAGE_NAMES` in the rollup config. Runs via `npm run test:dist`.
2. **`internal.rollup.config.mjs`** gets an `onwarn` handler that throws on `UNRESOLVED_IMPORT` instead of letting it slide. Error message names the three real causes (missing from node_modules, not built yet with a pointer to `turbo.json`, or misspelled). The same file exports `EXTERNAL_PACKAGE_NAMES` so the check script and rollup config stay in sync.
3. **`pull-request.yml` and `release.yml`** chain `npm run test:dist` into `additional-build-command`, which `auro-actions/.github/actions/build/action.yml` runs after `npm run build` and before `actions/cache@v4`. A failed check means the cache never saves; the matrix can't restore it; the release job's `needs: [build, test]` is unmet. Nothing publishes.

Three independent gates: build hard-fails, artifact verification catches anything that slips through, CI step ordering blocks publish on either failure.

## Lessons learned

- **`^build` in turbo silently resolves to `package.json` `dependencies`.** Any refactor that hoists deps to the root must also add an explicit `<name>#build` `dependsOn` block to `turbo.json` for each affected component. The hoist commit got this right for seven components and missed counter. There's no automated check that "package X's source imports workspace package Y" implies "turbo knows X depends on Y."
- **Rollup's default behavior on `UNRESOLVED_IMPORT` is wrong for library bundles.** Treating an unresolved bare import as an external is correct when downstream will rebundle. For a library that publishes its dist directly, the same default ships malformed artifacts. Set `onwarn` to throw on `UNRESOLVED_IMPORT` in every config that produces a publishable bundle.
- **Workspace tests certify the wrong thing.** WTR and framework tests pass against malformed dist whenever the workspace's `node_modules` happens to contain the leaked specifier. The only reliable certification is reading the artifact file from disk — what `test:dist` does.
- **`pull-request.yml` is not the publish gate; `release.yml` is.** PR coverage catches regressions before merge, but force-pushes, branch-protection bypasses, and direct-to-main flows skip it entirely. The artifact-contract test has to live on the publish path too.
- **Don't accept a proposed fix without reproducing the bug it claims to solve.** The `preferBuiltins` flip was plausible, sat at the right surface (rollup config), and got merged. It didn't fix anything. ~2 hours of investigation rolled it back. The cost of accepting a wrong fix isn't just the bad change — it's that the actual bug stays unfixed and the next investigator inherits a misleading prior.
