# Post-Mortem: Counter Shipped Unbundled `@aurodesignsystem/auro-dropdown` in 5.12.0/5.12.1 (AB#1575423)

**Branch:** `sunMota/rollup-fix`
**Scope:** Investigate the `@aurodesignsystem/auro-dropdown` and `@aurodesignsystem/auro-bibtemplate` imports that leaked into the published `auro-counter/dist/{index,registered}.js` in formkit 5.12.0 and 5.12.1. Verify the originally-proposed `preferBuiltins` fix, find the true root cause, and add defenses so the failure class cannot reach npm again.

## Reference Documents
- Affected publishes — `@aurodesignsystem/auro-formkit@5.12.0` (2026-05-08), `@aurodesignsystem/auro-formkit@5.12.1` (2026-05-11)
- Root-cause fix already on `dev` — [`310f18afb`](https://github.com/AlaskaAirlines/auro-formkit/commit/310f18afb) `ci: declare auro-counter build deps to prevent node-resolve race` (2026-06-04)
- Trigger commit — [`f623bd1cb`](https://github.com/AlaskaAirlines/auro-formkit/commit/f623bd1cb) `chore: reduce vulnerabilities, hoist shared deps, and remove unused packages` (2026-05-06)
- Initial (incorrect) fix on this branch — [`4f2273d08`](https://github.com/AlaskaAirlines/auro-formkit/commit/4f2273d08) `ci: fix packaging issue AB#1575423` (preferBuiltins flip)

## Symptoms → Lesson

| If you see... | This doc says... |
|---|---|
| A consumer reports `import { AuroDropdown } from '@aurodesignsystem/auro-dropdown'` at the top of a formkit dist file | Counter (or any component) is missing an explicit `<name>#build` `dependsOn` block in `turbo.json` AND has empty `dependencies` in its `package.json`. Turbo runs the dependent in parallel with what it depends on; rollup's nodeResolve catches a dead `dist/index.js` and silently externalizes. |
| `npm run build` reports `Tasks: 14 successful, 14 total` but `components/counter/dist/index.js` has a top-level `@aurodesignsystem/*` import | Build-order race condition. Wipe `dist/` and rerun — if the leak shows up on a clean build but not on a sequential single-component build, it's the race. |
| Reverting a config change (e.g. `preferBuiltins: true → false`) doesn't reproduce the bug | The config was never load-bearing. Walk back through `git log -- turbo.json package-lock.json components/<name>/package.json` and look for a commit that deleted a `dependencies` block. |
| Per-component WTR or framework smoke tests pass against a dist that has an unresolved bare import | Tests run inside the workspace, so the workspace symlinks resolve the leaked specifier. The failure is only visible to out-of-workspace consumers. Use `npm run test:dist` — it reads dist files from disk, not through a bundler. |
| Rollup emits `(!) Unresolved dependencies` to stderr and exits 0 | Default rollup behavior: warn, treat as external, continue. With this PR's `onwarn`, that warning now throws. |
| `npm ci` fails on the v5.12.0 tag with `lock file's undici@6.25.0 does not satisfy undici@6.26.0` | Separate lockfile drift in the tagged release. Not the cause of the bundle leak. `npm install` (lenient) reproduces the leak fine. |

## What Was Not in the Plan but Also Shipped

1. **`onwarn` error message names all three failure modes, not just "missing install".** Initial draft only mentioned `npm install` and `package.json` deps. The actual 5.12.0 failure was a build-order race where the dep WAS installed (workspace symlink existed) but its `dist/index.js` hadn't been written yet. Updated to enumerate: not installed, not built yet (with explicit pointer to `turbo.json` `<spec>#build` `dependsOn`), or misspelled. Future developers hitting this read the cause directly.
2. **JSDoc comment-stripping in `check-bundled-imports.mjs`.** First-pass regex matched bare-spec imports inside the bundled `date-fns` source's JSDoc examples (`* import { eoLocale } from 'date-fns/locale/eo'`). Three false positives reported on `datepicker/dist/index.js` triggered a side-quest. Fixed by stripping `/* ... */` and `// ...` before scanning, plus anchoring the regex to start-of-line.
3. **Identified the second leaked specifier.** Original ticket called out `auro-dropdown`. The 5.12.0 counter dist also has `import { AuroBibtemplate } from '@aurodesignsystem/auro-bibtemplate'` in `dist/index.js`. Same root cause; both deps were in the race window. The `registered.js` only leaks `auro-dropdown`.
4. **Confirmed the publishing cache cannot poison across runs.** Investigation initially suspected a cross-run cache poisoning vector. Reading `auro-actions/.github/actions/build/action.yml` showed the cache key is `build-${node-version}-${{ github.run_id }}` — scoped per run. The 5.12.0 bug happened entirely within a single workflow run: bad rollup output → cache saved → matrix passes (against in-workspace symlinks) → release job publishes. Removed cross-run cache as a hypothesis. Defenses tuned accordingly — gate inside the build step, before the cache save.
5. **`release.yml` had no `test:dist` matrix entry.** PR coverage caught it; release coverage didn't. Wired the same `additional-build-command` into both workflows. Build job runs the script after `npm run build`, before `actions/cache@v4` save. If the script fails, the cache never persists, the matrix can't restore, the release job's `needs: [build, test]` is unmet, and the publish step never runs.
6. **Lockfile drift visible at the v5.12.0 tag.** `npm ci` at that tag fails with `Invalid: lock file's undici@6.25.0 does not satisfy undici@6.26.0`. Not the cause of the bundle leak, but the broken lockfile suggests `npm ci` may have been falling back to `npm install` (or being skipped) in CI at the time. Filed as a follow-up.

## What the Plan Got Right

- The "fix packaging issue" framing was correct in spirit — the dist contract IS the contract that broke. The proposed code change (`preferBuiltins`) was wrong, but the surface area was right.
- The decision to investigate against a reproduction rather than ship the proposed fix sight-unseen caught the wrong fix before it landed in another publish.
- The split between "root cause fix" (turbo dependency graph, already on `dev` as `310f18afb`) and "defenses against the failure class" (this branch) lets each PR be reviewed by the right reviewer.

## What the Plan Missed

- **`^build` in turbo silently depends on `package.json` `dependencies` content.** Any refactor that hoists deps out of a component's `package.json` must also add an explicit `<name>#build` `dependsOn` block to `turbo.json`. The hoist commit [`f623bd1cb`](https://github.com/AlaskaAirlines/auro-formkit/commit/f623bd1cb) added turbo overrides for every other affected component (combobox, datepicker, select, form, checkbox, input, radio) but missed counter. There is no automated check for the consistency between "package X's source imports workspace package Y" and "turbo.json declares X#build depends on Y#build". Filed as `project_turbo-dependson-vs-imports`.
- **Rollup's silent-externalize default is wrong for library bundles.** Defaulting to "warn and treat as external" makes sense when rollup is used by a consumer that will rebundle. For a library that ships its dist directly to npm, the same default produces malformed published artifacts. Always set `onwarn` to throw on `UNRESOLVED_IMPORT` in rollup configs that produce final publishable bundles. The fix is one block of code per config; every formkit config that emits to `dist/` should have it.
- **Workspace tests mask out-of-workspace failures.** WTR loads from `src/`; framework smoke tests load `dist/` through vite, which resolves bare specifiers through the workspace's `node_modules`. Both pass against a dist that contains `import 'X'` as long as `X` exists somewhere in the workspace. Filed as `project_workspace-tests-mask-published-bundle-failures`. The only reliable check is reading the dist file from disk as a string — what `test:dist` does.
- **The publish path didn't have its own contract test.** `pull-request.yml` had no `test:dist` either, but PR coverage is inherently retroactive (catches regressions at the door, doesn't help with what already shipped). The bigger gap was that `release.yml` had no artifact-contract test — meaning a force-push, branch-protection bypass, or any path that didn't go through PR could ship a broken artifact unchallenged. Fixed by mirroring `additional-build-command` into both workflows.
- **The initial hypothesis was load-bearing for time spent.** ~2 hours of investigation went into validating that `preferBuiltins` wasn't the cause. The pattern: a plausible config-level fix gets proposed under deadline pressure; the next investigator inherits it as the prior. Acceptance of a proposed fix without a reproduction step is a class of bug independent of the technical one.

## Outcome

3 defenses landed on this branch, none of them touching the root-cause fix already on `dev`:

- `test/check-bundled-imports.mjs` — scans every `components/*/dist/{index,registered}.js` for bare-spec imports outside the lit allowlist. 26 files scanned in ~80ms.
- `packages/config/src/internal.rollup.config.mjs` — `onwarn` throws on `UNRESOLVED_IMPORT` with a 3-cause error message.
- `.github/workflows/{pull-request,release}.yml` — `additional-build-command: 'npx playwright install chromium && npm run test:dist'`. Gates the build job's cache save before the test matrix or publish can use it.

Verified end-to-end:
- Clean dist: `✓ bundled-imports check passed (26 files scanned)`.
- Injected `import { AuroDropdown } from '@aurodesignsystem/auro-dropdown'` at top of combobox dist: `✗ bundled-imports check failed (2 issues): ...` exit 1.
- Real `@aurodesignsystem/auro-formkit@5.12.0` tarball from npm: catches counter's two leaked specifiers exactly.
- Hidden `node_modules/@aurodesignsystem/auro-dropdown`, rebuilt combobox: rollup errors with the new `onwarn` message naming all three causes.
- `npm ci` clean install of v5.12.0 worktree, ran `npm install` then `npm run build` with fresh dist dirs: reproduced the leak with `Tasks: 14 successful, 14 total` and `dist/index.js` with both bare imports. Closed the loop.

The root cause is fixed on `dev` as of `310f18afb` (2026-06-04). This branch makes the same failure class — silent rollup externalization, regardless of trigger — impossible to publish.

## Recommendations for Future Bundle-Failure Investigations

1. **Reproduce before accepting a fix.** Pull the published tarball, scan the dist for the symptom, walk back through versions until clean. Match against `git log` for commits that touched dependency declarations or `turbo.json`. ~20 minutes of investigation rules out 80% of plausible-but-wrong hypotheses.
2. **When source imports a workspace sibling and the build is reproducible-but-slow, suspect turbo's dependency graph before the bundler config.** Check `turbo.json` for an explicit `<dependent>#build` `dependsOn` `<dependency>#build` entry. If missing, `^build` resolves to whatever's in `dependencies` of the dependent's `package.json` — which a hoisting refactor may have emptied.
3. **Don't trust in-workspace test results to certify a published bundle.** WTR and framework smoke tests will pass against a malformed dist as long as the workspace symlinks resolve the leaked specifier. Add an artifact-level check (`test:dist` or equivalent) that reads dist files from disk.
4. **Read `auro-actions` directly when reasoning about CI behavior.** The cache key (`build-${node-version}-${{ github.run_id }}`), step ordering (`additional-build-command` runs before `actions/cache@v4` save), and matrix gating (`release` `needs: [build, test]`) are all visible in `.github/actions/build/action.yml` and `.github/workflows/{pull-request,release}.yml`. ~5 minutes of reading rules out cross-run cache poisoning and other ghost hypotheses.
5. **Set rollup `onwarn` to throw on `UNRESOLVED_IMPORT` in every config that produces a publishable artifact.** This is a one-time, low-cost change that closes the silent-externalize failure mode for all future configs derived from the affected one.
