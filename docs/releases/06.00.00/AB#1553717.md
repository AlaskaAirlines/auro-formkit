# AB#1553717 тАФ Datepicker Calendar Performance & Accessibility Postmortem

**Branch:** `jbaker/datepickerPerformance`  
**PR:** [#1480](https://github.com/AlaskaAirlines/auro-formkit/pull/1480)  
**Scope:** 46 commits, 34 files changed, ~3,500 insertions across `auro-calendar`, `auro-calendar-cell`, `auro-calendar-month`, `auro-datepicker`, tests, docs, and styles

## 1. Starting Point: The Performance Problem

On `dev`, every arrow-key press or mouse hover in the calendar triggered a full Lit reactive lifecycle across all 28тАУ42 cells in a month. A single keystroke caused 35+ synchronous re-renders, each instantiating `Intl.DateTimeFormat`, querying the DOM for slot content, and re-evaluating 10+ conditional CSS classes. This saturated the browser's rendering pipeline so badly that screen reader announcements were delayed, dropped, or arrived out of order.

A Technical Research Document (`docs/datepicker-calendar-performance.md`) was written to evaluate four approaches: optimizing the existing lifecycle (Option A), a live region announcement pattern (Option B), virtual rendering (Option C), and Web Workers (Option D). Option B was selected as the recommended path because it eliminated the root cause тАФ the coupling between cell rendering and SR announcements тАФ rather than optimizing around it.

## 2. Phase 1: Keyboard Navigation & Active Cell Management

### What was built (manual + AI)

The first phase implemented keyboard navigation on `#calendarGrid`. AI generated the initial scaffolding for `handleGridKeyDown()`, `setActiveCell()`, `computeActiveDate()`, and the live region announcement pipeline. Manual development defined the behavioral requirements: which cell should receive focus on open, how blackout cells participate in navigation, and how month boundaries work.

### Key decision: roving tabindex removal

The `dev` branch used a traditional roving tabindex pattern тАФ the focused cell's button received `tabindex="0"` and all others `tabindex="-1"`. This required DOM focus to physically move to each cell button inside nested shadow DOMs (calendar тЖТ month тЖТ cell). We moved to a model where `#calendarGrid` holds DOM focus permanently (`tabindex="0"`) and a debounced `aria-live` region announces the active cell to screen readers.

**Why:** With nested shadow DOM, moving DOM focus between cells means crossing three shadow boundaries per keystroke. The browser must retarget focus events at each boundary, fire `focusin`/`focusout` cascades, and update the accessibility tree for the newly focused element тАФ all within a single frame. With the live region pattern, focus never moves; only the live region's `textContent` changes. This is a single DOM write instead of a shadow DOM focus traversal.

### Decision path that evolved during development

`computeActiveDate()` went through multiple iterations:

1. **First pass (AI):** Priority order was selected date тЖТ today тЖТ scan forward тЖТ scan backward. This worked for simple cases but failed when `centralDate` pointed to a month far from today тАФ the active cell landed on today's date, which had no corresponding DOM cell in the rendered month.

2. **Manual correction:** Added `centralDate`-aware scanning that constrains the initial search to the visible month(s) before falling back to the broader scan. This ensures the active cell always has a matching DOM element.

3. **AI refinement:** Added handling for edge cases тАФ `minDate`/`maxDate` far from today, all-blackout months (fall back to in-range even if blackout), unbounded min with finite max (birth-date picker pattern). Each was driven by a failing test case that manual development identified.

## 3. Phase 2: The Dual-Handler Bug & Month Navigation

### Problem discovered during testing

After moving keyboard handling from `auro-calendar-month` to `auro-calendar`, both components were processing the same `keydown` events. The month had `@keydown="${this.handleGridKeyDown}"` on its `.table` div, and the calendar had `@keydown` on `#calendarGrid`. Because `keydown` events are composed and bubble through shadow DOM, a single keystroke fired both handlers, causing double navigation.

**AI diagnosis, manual verification:** AI identified the dual-handler issue from the test failure output. The fix was removing the `@keydown` binding from the month's `.table` div. Manual review confirmed that the month's `handleGridKeyDown` method became dead code after this change.

### Month navigation and `updateActiveCellForVisibleMonth()`

When clicking the prev/next month buttons, the active cell needed to update to a cell in the newly visible month. This went through several iterations:

1. **Keep-if-rendered:** If the current active cell was still in the DOM after month change, keep it. Otherwise pick the first enabled cell. Failed because in a 2-month desktop view, the previous month's cells are still rendered тАФ the active cell never moved.

2. **Check-blackout:** Always move to first non-blackout cell. Failed edge case: what if the entire month is blacked out?

3. **Always first enabled cell (final):** After month navigation, always call `computeActiveDate({ skipDateFrom: true })` to find the first enabled cell in the visible months. The `skipDateFrom` flag prevents jumping back to the selected date's month. AI proposed this simplification after the third test failure. Manual review agreed it matched the expected UX: "I clicked Next, I should be looking at the new month."

### Timing: double-rAF pattern

`updateActiveCellForVisibleMonth()` initially used `this.updateComplete.then(...)` to wait for child cells to render. This resolved before child components had finished their own render cycles and cached their button references (`_cachedButton`). The active cell's button was null when `setActiveCell()` tried to set `tabindex`.

**Manual discovery, AI fix:** Manual testing surfaced the timing failure. AI proposed the double-`requestAnimationFrame` pattern тАФ the outer rAF fires after the browser's next paint, the inner rAF fires after child components have completed their render. This became the standard timing pattern used throughout the calendar's async operations.

## 4. Phase 3: Visual Focus Management

### Problem: focus ring showing on two cells simultaneously

**Repro (manual):** Open blackout-label demo тЖТ click Next тЖТ Tab into grid (Feb 1 gets focus) тЖТ ArrowRight to Feb 2 тЖТ both Feb 1 and Feb 2 show the gray focus ring.

**Root cause analysis (AI + manual):** Two CSS rules produced the ring: `:focus-visible` (native browser focus on the cell button) and `.visualFocus` (imperatively managed class). When the user Tabbed into the grid, DOM focus landed on the cell button (which had `tabindex="0"` under the old roving pattern). ArrowRight moved `visualFocus` to the new cell via `setActiveCell()`, but DOM focus stayed on the old button. Both cells showed the ring тАФ one from `:focus-visible`, one from `.visualFocus`.

### First attempt: focus redirect (failed)

AI proposed intercepting `focusin` on `#calendarGrid` and redirecting focus from descendant cell buttons back to the grid wrapper. The `handleGridFocusOut` was also modified to check `event.relatedTarget` to avoid clearing `visualFocus` during the redirect.

**Result:** Tests timed out with 0 passed, 0 failed тАФ an infinite focus loop. The redirect triggered a new `focusin`, which triggered another redirect. The guard (`event.target !== gridWrapper`) should have prevented this, but Lit's event retargeting through shadow DOM boundaries meant `event.target` was not always the grid wrapper even when the grid had focus.

### Second attempt: CSS-only fix (adopted)

**Manual decision:** Instead of fighting shadow DOM focus retargeting, remove `:focus-visible` from the cell button CSS entirely. The `visualFocus` class is the sole mechanism for showing the focus ring, controlled imperatively by `setActiveCell()` and `handleGridFocusIn()`/`handleGridFocusOut()`.

Changes:
- `style-auro-calendar-cell.scss`: Replaced `&:focus-visible, &.visualFocus { ... }` with `&:focus-visible { outline: none; background-color: unset; }` and `&.activeCell { ... }` separately.
- `color-cell.scss`: Removed `&:focus-visible:not(.selected)` from the background-color rule. Only `.activeCell:not(.selected)` applies the background on focus.

This eliminated the dual-ring problem without any JavaScript focus manipulation.

### `_cachedButton` staleness

`clearActive()` originally only used `this._cachedButton` to remove `visualFocus`. But `_cachedButton` is set asynchronously in `connectedCallback` and `updated()` via `this.updateComplete.then(...)`. Between renders it could be null, leaving stale `visualFocus` classes on buttons.

**AI fix:** Added `|| this.shadowRoot.querySelector('button.day')` fallback to both `setActive()` and `clearActive()`.

## 5. Phase 4: Tab Order Correction

### Problem: Tab from grid goes to prev month button

After focus management was working, Tab from the calendar grid went to the prev month navigation button instead of the Done button.

**Root cause (AI analysis, manual verification):** `#calendarGrid` had `tabindex="-1"` (focusable programmatically, not a tab stop). The active cell button had `tabindex="0"` (a tab stop inside nested shadow DOM). When Tab was pressed from `#calendarGrid`, the browser found the cell button as the next tab stop тАФ an invisible focus move. The next Tab exited the cell's shadow DOM and landed on the prev month button.

**Fix:** Two complementary changes:
1. `#calendarGrid` changed from `tabindex="-1"` to `tabindex="0"` тАФ making it a proper tab stop.
2. All cell buttons changed to always render with `tabindex="-1"` тАФ they are never direct tab stops.

`setActive()` and `clearActive()` no longer toggle `tabindex` on buttons (the roving tabindex pattern was fully removed). The `active` property and `.activeCell` class are the only things these methods manage.

The roving tabindex test was updated to verify the new behavior: all buttons at `tabindex="-1"`, `active` property tracks the active cell.

## 6. Phase 5: Cross-Browser Screen Reader Consistency

### Problem: Chrome announcements inconsistent across machines

VoiceOver + Safari worked consistently on all test machines. VoiceOver + Chrome worked on one Mac but not another running the same OS and Chrome version.

### Attempt 1: aria-activedescendant with static ID (failed)

The initial implementation used `aria-activedescendant="activeDescendantProxy"` pointing to a proxy `<span>` in the calendar's shadow root. When the user navigated cells, `setActiveCell()` updated the proxy's `aria-label` but the `aria-activedescendant` value never changed. Chrome requires the attribute value itself to change to fire its `ActiveDescendantChanged` accessibility event. Safari monitors the referenced element's attributes directly, which is why it worked everywhere.

### Attempt 2: toggle the attribute (failed)

AI proposed removing and re-setting `aria-activedescendant` on each navigation:
```js
gridWrapper.removeAttribute('aria-activedescendant');
gridWrapper.setAttribute('aria-activedescendant', 'activeDescendantProxy');
```

**Still failed on some Chrome instances.** Chrome batches synchronous DOM mutations within the same microtask. The `removeAttribute` + `setAttribute` with the same string value was coalesced into a no-op тАФ Chrome saw no net change.

### Attempt 3: unique proxy IDs (partially worked)

Each `setActiveCell()` call incremented a counter and gave the proxy `<span>` a new unique `id` (`activeDescendantProxy-1`, `-2`, `-3`, ...). Chrome now saw a genuinely different `aria-activedescendant` value on every navigation.

**Still inconsistent.** Testing across multiple machines revealed that even with unique IDs, Chrome's `aria-activedescendant` support in shadow DOM was unreliable. The proxy `<span>` was in the same shadow root as `#calendarGrid`, which should satisfy the spec, but Chrome's accessibility tree updates were still being dropped on some machines.

### Final fix: remove aria-activedescendant entirely (adopted)

**Manual decision:** After three failed attempts to make `aria-activedescendant` work consistently in Chrome, the entire mechanism was removed. The proxy `<span>`, the `_proxyIdCounter`, the `data-role` attribute for locating the proxy, and the `aria-activedescendant` attribute on `#calendarGrid` were all deleted.

The calendar now relies exclusively on an `aria-live="assertive"` region appended directly to the dropdown's `<dialog>` element for all SR announcements during cell navigation. The live region uses a debounced announcement pattern (`announceFocusDebounced()` with a 150ms delay) so rapid arrow-key navigation only announces the final cell rather than queueing stale announcements.

**Why this works reliably:** `aria-live` regions are one of the oldest and most consistently implemented accessibility features across all browsers and screen readers. Unlike `aria-activedescendant` тАФ which requires the browser to resolve an ID reference across the accessibility tree and fire a specific event тАФ `aria-live` simply monitors a DOM element for `textContent` changes and announces them. There is no ID resolution, no shadow DOM boundary complexity, and no browser-specific event firing behavior to contend with.

**Trade-off:** The `aria-activedescendant` pattern would have provided richer semantic information тАФ the SR knows the focused element is a gridcell with `aria-selected`, `aria-current`, and `aria-disabled` states. With the live region, all of this context is flattened into a single announcement string built by `buildFocusAnnouncement()`. This is less semantically precise but functionally equivalent: the user hears the same information (date, selection state, blackout status, range position) on every navigation.

## 6a. Phase 5a: Live Region Placement & Instance Isolation

### Problem: shared singleton caused cross-instance conflicts

The initial live region implementation used a single shared `<div>` on `document.body` with a fixed `id="auro-calendar-live-region"`. This worked for single datepicker pages but failed when multiple datepicker instances existed тАФ announcements from one instance could overwrite another's, and cleanup in `disconnectedCallback` had to count remaining instances before removing the element.

### Attempt 1: datepicker light DOM (failed)

Moved the live region into the datepicker's light DOM via `this.getRootNode()?.host?.appendChild(el)`. The datepicker has no default `<slot>`, only named slots тАФ so the unslotted light DOM element was excluded from the rendered (flat) tree. Screen readers follow the flat tree and never saw the live region.

### Attempt 2: datepicker shadow DOM (failed)

Appended the live region to the datepicker's shadow root via `this.getRootNode().appendChild(el)`. Two problems:
1. The calendar's `getRootNode()` actually returns the **dropdown's** shadow root (the component hierarchy is `datepicker тЖТ dropdown тЖТ calendar`), so the live region ended up in the dropdown's shadow DOM, not the datepicker's.
2. Even if correctly placed in the datepicker's shadow DOM, Chrome still does not reliably observe `aria-live` regions inside shadow DOM тАФ the same fundamental issue that drove the `aria-activedescendant` removal.

### Attempt 3: document.body with per-instance unique IDs (superseded)

Returned the live region to `document.body` тАФ the only universally reliable location тАФ but gave each datepicker instance its own uniquely-identified region:

- `_getLiveRegionId()` generates `auro-calendar-live-region-{suffix}` where the suffix is either the datepicker host's `id` attribute or a random 8-character hash.
- Each calendar creates and manages its own region independently тАФ no shared singleton, no cross-instance conflicts.
- `disconnectedCallback()` simply removes its own region without checking for other instances.

### Race condition: bib open vs. month navigation

After the unique-ID refactor, month navigation announcement tests failed because the cell date appeared in the live region instead of the month name. Root cause: `focusActiveCellWhenReady()` (triggered on bib open) uses a retry loop of up to 20 `requestAnimationFrame` calls to find rendered cells. When the test clicked the month nav button before this rAF chain fully settled, the bib-open announcement (`announceSelection` with the initial cell date) fired *after* the month navigation announcement, overwriting it.

**Two fixes applied:**

1. **`announceSelection()` cancels previous pending rAF** тАФ stores the rAF ID in `this._announceRafId` and calls `cancelAnimationFrame()` before queuing a new one. When `focusActiveCellWhenReady` and `announceMonthChange` race, only the last call's announcement lands.

2. **`disconnectedCallback()` cancels all pending timers** тАФ both `_announceRafId` (via `cancelAnimationFrame`) and `_focusAnnounceTimer` (via `clearTimeout`). This prevents a disconnected calendar's pending callbacks from firing, recreating an orphaned live region on `document.body` that would interfere with subsequent test fixtures.

### Phase 5a discovery: `document.body` is inert inside modal dialogs

After the per-instance unique ID approach appeared to work in unit tests, real-device testing with VoiceOver revealed that **no cell navigation was being announced** тАФ not just on open, but at all. The live region existed on `document.body` and its `textContent` was updating correctly (unit tests confirmed this), but VoiceOver never spoke the announcements.

**Root cause:** The dropdown uses `dialog.showModal()` for fullscreen mode, which promotes the dialog to the browser's **top layer** and makes everything outside it inert тАФ including the live region on `document.body`. On desktop, `_setPageInert()` explicitly walks the DOM tree and sets `inert=true` on all siblings of the datepicker host, which also captures the live region. In both modes, the live region was programmatically hidden from the accessibility tree.

Unit tests passed because they check DOM `textContent`, not actual screen reader output. The `inert` attribute doesn't prevent JavaScript from reading or writing `textContent` тАФ it only removes the element from the accessibility tree.

### Attempt 4: live region in the calendar's shadow DOM template (superseded)

Moved the live region from `document.body` into the calendar component's Lit `render()` template as a visually-hidden `<div id="calendarLiveRegion" aria-live="assertive" aria-atomic="true">`. Because the calendar is slotted into the dropdown's `<dialog>`, this element lives inside the dialog's accessible scope in both fullscreen (`showModal()`) and desktop modal (`_setPageInert()`) modes.

This solved the inert problem but introduced a new one: **Chrome inconsistently observes `aria-live` mutations in deeply nested shadow DOM across machines and versions.** The calendar's shadow DOM is nested two+ levels deep (dropdown bib тЖТ slot тЖТ calendar тЖТ shadow root), and Chrome's accessibility tree observation at that depth proved unreliable тАФ the same cross-machine inconsistency seen with `aria-activedescendant`. VoiceOver + Safari worked consistently, but VoiceOver + Chrome worked on some machines and not others.

### Final approach: live region appended to the dropdown's `<dialog>` element (adopted)

Moved the live region out of the calendar's shadow DOM template and into the dropdown bib's `<dialog>` element via imperative DOM manipulation. `getOrCreateLiveRegion()` lazily creates a `<div>` and appends it as a direct child of the dialog element in the dropdown bib's shadow DOM.

This placement is optimal for two reasons:

1. **Inside the dialog's accessible scope** тАФ `dialog.showModal()` and `_setPageInert()` cannot inert it.
2. **Only one shadow root deep** тАФ the dialog is in the dropdown bib's shadow DOM, which Chrome handles reliably. The calendar's own shadow DOM (nested inside the bib via slotting) is two+ levels deep and unreliable for `aria-live` observation in Chrome.

Implementation details:
- `getOrCreateLiveRegion()` accesses the dialog via `this.dropdown.bibContent.shadowRoot.querySelector('dialog')` and appends a `<div>` with a unique ID (`auro-calendar-live-{instanceId}`).
- `_calendarInstanceId` is set to `Date.now().toString(36)` in the constructor тАФ no `Math.random()` for security compliance.
- `disconnectedCallback()` removes the live region element and cancels pending rAF/timer handles.
- Null guard in `announceSelection()` handles the case where the dialog isn't available yet (e.g., during initial render before the dropdown is connected).

### Announcement timing: double-rAF

`announceSelection()` uses a double-`requestAnimationFrame` pattern: the live region's `textContent` is cleared synchronously, then set two animation frames later. Chrome batches accessibility tree mutations within a single animation frame, so a same-frame clear+set can be coalesced into a no-op if the new value matches a recently announced string. Splitting across two frames guarantees Chrome sees two distinct tree states and fires a new accessibility event for the content change.

### Initial cell announcement timing

`focusActiveCellWhenReady()` calls `focusActiveCell()` (which moves DOM focus to `#calendarGrid`) and then `announceSelection()`. The announcement is delayed by 500ms via `setTimeout` so it arrives after VoiceOver finishes speaking the focus-change announcement for the grid wrapper. Without this delay, VoiceOver drops the live region update because it's already mid-announcement from the focus move.

### Landmark pollution: removing `role="status"`

The initial live region included `role="status"` alongside `aria-live="assertive"`. This caused VoiceOver to announce "and one more item" when the bib opened тАФ VoiceOver counts landmarks on the page and the status role registered as an additional landmark. Removing `role="status"` eliminated the extra announcement. `aria-live="assertive"` alone is sufficient to trigger SR announcements on content changes; the `role` attribute is unnecessary and counterproductive.

### VoiceOver cursor highlights the grid, not individual cells

With `aria-activedescendant` removed, VoiceOver's visual cursor (black rectangle) highlights the entire `#calendarGrid` rather than individual cells during keyboard navigation. This is an inherent trade-off of the `aria-live` approach: DOM focus stays on the grid wrapper, and VoiceOver draws its cursor around the focused element. Moving DOM focus to each cell button (roving tabindex) would restore per-cell highlighting but reintroduces the performance and shadow DOM focus issues that drove the current architecture. The `.activeCell` CSS class provides the visual indicator of which cell is active, making the VoiceOver cursor position cosmetic rather than functional.

## 6b. Phase 5b: CSS Class Naming & Cell Styling

### Class rename: `visualFocus` тЖТ `activeCell`

The `.visualFocus` class was misleadingly named тАФ it tracked the keyboard-navigated active cell, not browser focus state. It was applied when a cell became the active target of arrow-key navigation, regardless of whether the grid had DOM focus. The class was renamed to `.activeCell` across all source files to accurately reflect its semantic purpose:

- `style-auro-calendar-cell.scss`: `&.activeCell` outline rule
- `color-cell.scss`: `&.activeCell:not(.selected)` background and border rules
- `style-auro-calendar.scss`: comment updated
- `auro-calendar.js`: `classList.add/remove('activeCell')` in `setActiveCell()`, `handleGridFocusIn()`, `handleGridFocusOut()`
- `auro-calendar-cell.js`: `clearActive()` removes `activeCell`

### Active cell styling

The `.activeCell` styling was refined to use design token-driven colors:

- `style-auro-calendar-cell.scss`: 2px solid outline with -4px offset (inset, inside the cell border)
- `color-cell.scss`: `&.activeCell:not(.selected)` sets `border-color` to `--ds-advanced-color-state-selected` (Alaska blue) and `outline-color` to the cell container color, creating a visible active indicator that contrasts with both the cell background and the selected state

### `.buttonWrapper` vertical centering

Cell button content (date number and date slot) was not vertically centered within the 48├Ч56px button. Added flexbox to the base `.buttonWrapper` rule:

```scss
.buttonWrapper {
  display: flex;
  flex-direction: column;
  align-items: center;
  justify-content: center;
  height: 100%;
}
```

The `height: 100%` ensures the wrapper fills the button so `justify-content: center` has room to distribute space.

## 7. What AI Did Well

- **Scaffolding large methods:** `computeActiveDate()`, `handleGridKeyDown()`, `handleMonthBoundary()`, and the announcement pipeline were all initially generated by AI and required relatively few corrections.
- **Pattern recognition:** Identified the dual-handler bug from test output, proposed the double-rAF timing pattern, and diagnosed the Chrome `aria-activedescendant` batching issue.
- **Test adaptation:** Rewrote tests to target `#calendarGrid` instead of month-level grids, updated roving tabindex assertions, and adapted live region queries through three placement changes (light DOM тЖТ shadow DOM тЖТ document body with unique IDs).
- **Edge case enumeration:** Generated comprehensive `computeActiveDate` fallback chains covering min/max far from today, all-blackout months, and unbounded ranges.
- **Race condition diagnosis:** Identified the `focusActiveCellWhenReady` vs. `announceMonthChange` race from test failure output (cell date appearing instead of month name) and proposed the `cancelAnimationFrame` guard pattern.

## 8. Where Manual Intervention Was Required

- **Behavioral specification:** AI did not know what the correct UX should be тАФ which cell should be active on open, whether blackout cells are skippable or focusable, how month boundaries interact with range mode. These were all manual decisions.
- **Focus redirect failure:** AI's initial approach (intercepting `focusin` and redirecting DOM focus) caused an infinite loop. The CSS-only alternative was a manual decision based on understanding that the `visualFocus` class was already the intended mechanism.
- **aria-activedescendant abandonment:** AI proposed three successive fixes for Chrome's inconsistent `aria-activedescendant` behavior (toggle attribute, unique IDs, data-role lookup). All were technically sound but failed due to Chrome's unpredictable shadow DOM accessibility tree behavior. The manual decision to abandon `aria-activedescendant` entirely in favor of `aria-live` was based on real-device testing across multiple machines тАФ something AI could not do.
- **Shadow DOM event retargeting:** AI repeatedly underestimated how `event.target` and `event.relatedTarget` behave across shadow DOM boundaries. Checks like `event.target !== gridWrapper` don't work as expected when the event crosses nested shadow roots with different retargeting rules.
- **Timing sensitivity:** Every async operation (month navigation, bib open, cell rendering) required manual verification of the timing pattern. AI's default instinct was `await this.updateComplete`, which consistently failed for cross-component waits in nested shadow DOM.

## 9. Lessons for Future AI-Assisted Development

1. **Shadow DOM is adversarial to focus management.** Native browser APIs (`focus()`, `focusin`, `focusout`, `relatedTarget`) behave unpredictably across nested shadow roots. Prefer patterns that avoid moving DOM focus between shadow trees entirely.

2. **`aria-activedescendant` is unreliable in shadow DOM.** Despite being spec-defined, Chrome's implementation of `aria-activedescendant` in shadow DOM is inconsistent across machines and versions. The `aria-live` region pattern is older, simpler, and universally reliable. For components with nested shadow DOM, prefer `aria-live` over `aria-activedescendant`.

3. **Synchronous DOM mutations can be coalesced.** Removing and re-adding the same attribute value in the same microtask is not guaranteed to produce two accessibility events. Even using genuinely different values may not work if the browser's accessibility tree update is batched or deferred.

4. **AI excels at code generation but not at behavioral design.** The most productive workflow was: manual developer defines the expected behavior, AI generates the implementation, manual developer tests and identifies edge cases, AI fixes them. Attempting to have AI define behavior led to plausible but incorrect assumptions.

5. **Double-rAF is the reliable cross-component timing primitive for Lit.** `updateComplete` only waits for the current component's render. For operations that depend on child components in nested shadow DOMs having finished their render and cached their DOM references, `requestAnimationFrame(() => requestAnimationFrame(() => { ... }))` is the reliable pattern.

6. **When a spec feature doesn't work, cut it.** Three iterations of workarounds for `aria-activedescendant` added complexity (proxy elements, ID counters, data-role attributes) without solving the problem. The final fix was removing all of it and using the simpler, proven `aria-live` mechanism. Sunk cost in workaround code should not prevent choosing the fundamentally different approach.

7. **`aria-live` region placement requires both dialog scope and minimal shadow DOM depth.** `dialog.showModal()` makes everything outside the dialog inert, and `_setPageInert()` explicitly inerts siblings тАФ so the live region must be inside the dialog. But Chrome also inconsistently observes `aria-live` mutations in deeply nested shadow DOM (2+ levels). The reliable placement is a direct child of the `<dialog>` element itself (one shadow root deep in the dropdown bib), not in the calendar's shadow DOM template (two+ levels deep via slotting). This combines the "inside the modal" requirement with the "minimal shadow DOM nesting" requirement.

8. **Async announcement pipelines need cancellation.** When multiple code paths can write to the same live region (`focusActiveCellWhenReady`, `announceMonthChange`, `announceFocusDebounced`), each path must cancel pending writes from previous paths. Without `cancelAnimationFrame` and `clearTimeout` guards, race conditions between bib open, month navigation, and cell focus produce stale or incorrect SR announcements.

9. **`disconnectedCallback` must cancel all pending async work.** A pending `requestAnimationFrame` or `setTimeout` that fires after `disconnectedCallback` can recreate DOM elements (like live regions) that were just cleaned up. Always cancel rAF IDs and timer handles during teardown to prevent orphaned DOM nodes and test pollution.

10. **Use semantic class names that reflect purpose, not mechanism.** The `.visualFocus` class was renamed to `.activeCell` because it tracked the keyboard-navigated active cell, not browser focus. Misleading names cause confusion when debugging and make the codebase harder to reason about тАФ especially when both "focus" and "active" have distinct DOM meanings.

11. **`role="status"` creates a landmark.** Adding `role="status"` to a live region registers it as a status landmark in the accessibility tree. VoiceOver counts landmarks and announces extras ("and one more item"). Use `aria-live` alone when you only need announcement behavior without landmark semantics.
